## 0. 文档定位

### 0.1 这份文件是什么

- 9 个候选模型的**形态级**讨论稿:命名、字段类别、依赖关系、待调研项、回退条件
- 每个模型用统一 6 维标签([稳][研][赖][敏][史][先])与 8 子项模板,便于评审会逐模型决议(进入 V2 / 延后到 V2.x / 砍掉)
- 跨模型模式(字典引用 / users-members 关联 / 软删 / 审计 / 附件归属 / events 回退)的候选方案与触发回退条件

### 0.2 这份文件不是什么

- **不是**最终设计 — 任何"已确认"措辞限于 baseline / research 已锁部分,本草案不出新结论
- **不是** Prisma schema — 全文不写 `@id` / `@@map` / `@relation` 等 Prisma DSL
- **不是** API 契约 — 全文不写 `/api/...` 路径 / HTTP 方法 / 入参出参
- **不是**字段定义 — 全文**不写字段类型**(`String` / `Int` / `DateTime` 等),只写"字段类别"
- **不是**开发任务 — 不出现"实现 / 创建 / migration / seed 落地"等执行词

### 0.3 严守的边界

- **不写**真实部门名 / 真实等级取值 / 真实活动类型 / 真实证书类型 / 真实字典内容(对应 `research.md §7-R13` / `CLAUDE.md §18.2`)
- **不写**业务字段的具体名字(只允许写 baseline §2.1 已锁的通用字段:`id` / `createdAt` / `updatedAt` / `deletedAt` / `status`);业务字段一律以**类别词**描述
- **不重复** baseline 已锁的规则(本文 §5 给出"不再重复讨论"清单)
- **不**把候选措辞写成最终设计;每个候选必须配"备选方案 / 回退条件",对应 `CLAUDE.md §18.3`
- **不**实现 `research.md §3` 任一暂不做项

### 0.4 修订纪律

- 修订需用户拍板,**禁止** AI 自行扩张或自行回填 D5 结论
- 修订 commit message 前缀:`v2-design: <章节> <简述>`
- 修订需在附录版本表显式记录

## 1. 标注体系

每个候选模型在节首必须标注以下 6 维标签,**不允许空标签**:

| 维度 | 含义 | 判定准则 |
|---|---|---|
| **[稳]** | 形态稳定 | 命名、最小字段集已是共识,只待字段细节决策 |
| **[研]** | 需要调研 | 仍需 D5 / 草案评审 / 真实数据访谈才能定型 |
| **[赖]** | 依赖模块 | 列出本模型未来会被哪些模块引用(指向草案后续模型 / 未来 V2 业务模块) |
| **[敏]** | 敏感信息 | 是否承载个人敏感信息;若是,**强制**走 `research.md §4.3` 三问(业务用途 / 查看角色 / 保存期限) |
| **[史]** | 历史 / 版本化 | 是否需要保留版本或变更历史(与 `research.md §4.4` 联动) |
| **[先]** | 第一阶段 | 是否建议进入 V2 第一阶段实现(`是` / `否` / `延后` / `待调研`) |

### 1.1 标注用法

- **同维多值**:用 `/` 分隔(例 `[赖:members / member_profiles]`)
- **空集合**:写明`无`,**禁止**留空
- **不确定**:用 `[?]` 后缀(例 `[先:?]`),并在 §3.x.6"进入实现前必须确认"中显式回答需要哪些前置才能去掉 `?`
- **非业务字段**:`[敏:否]` / `[史:否]` 等显式否定形式必填

### 1.2 标签 vs 草案细化的关系

- 标签是**摘要级**,§3.x 子节是**展开级**;评审时先看标签速览,再展开评估
- 标签**不是**最终决议;评审通过后由 D7 把"建议进入第一阶段"标记锁定为"实现 / 延后 / 砍掉"
- 标签的演化记录在附录版本表

## 3. 候选模型逐个展开

### 3.1 `dictionaries`

> **标签**:[稳] / [赖:几乎所有业务表] / [敏:否] / [史:否(items 用启停标记 + 防御性 deletedAt 待评估)] / [先:是]

#### 3.1.1 解决什么问题

把"可变业务枚举"(队员等级 / 资质 / 部门类型 / 活动类型 / 证书类型 / 装备类别等)从代码硬编码迁出,改由数据维护。代码侧只持有"字典 code"引用,不持有具体取值。

**红线**(`research.md §6.3`):系统级稳定枚举(`Role` / `UserStatus`)**不**入字典,继续走 Prisma enum。

#### 3.1.2 当前倾向

倾向**双表**(`dict_types` + `dict_items`):前者承载类型元数据(类型名 / 是否树形 / 是否可停用 / 是否可运营自助新增),后者承载具体取值。

**备选方案**:单表 `dictionaries(type, code, label, sortOrder, parentCode?, status, metadata?)`。
**回退条件**:若 D5 §4.8 调研显示所有字典都是同质扁平结构(无树形 / 无类型元数据需求 / 字典类型 ≤3),回退到单表。

#### 3.1.3 候选字段类别(不写字段名,不写类型)

**dict_types 形态**:

- 通用字段:`id` / `createdAt` / `updatedAt` / `deletedAt`(baseline §2.1 已锁)
- 类型标识类
- 类型元数据类(是否树形 / 是否运营可改 / 是否多语言 等,具体取值待 §4.8 调研)
- 启停状态类(`status` 走 enum,baseline §2.2.3)

**dict_items 形态**:

- 通用字段:`id` / `createdAt` / `updatedAt` / `deletedAt`
- 类型外键类(指向 dict_types)
- 业务标识类(items 自身的 `code`,业务侧引用)
- 业务展示类(label / 排序权重)
- 父子关系类(若类型为树形 — 父级外键,自我引用)
- 启停状态类
- 扩展属性类(metadata,具体形态待 §4.8 调研)

#### 3.1.4 依赖哪些已确认基线

- baseline §1.1 段位 `120xx + 121xx`
- baseline §2.1 通用命名(`id` / `createdAt` / `deletedAt`)
- baseline §2.2.3 启停字段统一 `status`
- baseline §6 错误码命名(如 `DICTIONARY_NOT_FOUND` / `DICT_ITEM_CODE_ALREADY_EXISTS`)
- baseline §10 软删除显式封装

#### 3.1.5 待调研问题(指向 research.md)

- `research.md §4.8`:类型元数据是否真有用 / 字典异质性程度 / 单表 vs 双表的成本权衡
- `research.md §5.1` 红线:仓库 seed **不**写真实字典内容,只用 neutral-demo 占位

#### 3.1.6 进入实现前必须确认什么

- §4.8 调研出阶段性结论(单表 vs 双表)
- 字典引用方式(§4.1 跨模型模式)选边
- 字典 items 的删除策略(物理删 vs 仅启停 + 防御性 `deletedAt`)逐字典评估

#### 3.1.7 设计错了,后续代价是什么

- **迁移成本**:中(若选错单/双表,迁移涉及全部业务表的字典外键改写)
- **历史数据修复成本**:低(字典内容由运营录入,不涉及历史业务数据迁移)
- **后续模块耦合成本**:**高**(几乎所有 V2 业务模块引用字典,字典模型决策错会扩散到所有引用方)

#### 3.1.8 是否建议进入第一阶段

**是**(基建优先级最高 — 多个业务模型都依赖字典 code)。但**前置**:§4.8 调研结论 + §4.1 字典引用方式选定。

### 3.2 `organizations`

> **标签**:[研] / [赖:members / member_departments / events / 未来业务模块] / [敏:否] / [史:部分 — 节点改名/移动/合并历史是否保留待 §4.1 调研] / [先:是(最小骨架)]

#### 3.2.1 解决什么问题

提供"组织树"这一形态,**不**预设具体节点。所有节点由运营在评审通过后录入(seed 仅用 neutral-demo 占位)。覆盖业务上的"部门 / 小组 / 临时编组"等多种语义类别。

**红线**(baseline §2.2.2.1):`organizations` 是**技术模型名**(覆盖所有树节点);`department` 是**业务语义别名**,只在已确认的业务命名上下文使用(中间表 / 业务字段);**禁止**同项目混用两套主表名。

#### 3.2.2 当前倾向

倾向**单根树**(顶层一个根节点),节点用 `parentId` 自我引用。具体节点类别(部门 / 小组 / 临时编组)通过节点上的"类别字典 code"区分,**不**拆表。

**备选方案**:多根树(顶层并列多个根节点);职能线 + 业务线两套并列树。
**回退条件**:若 D5 §4.1 Q1.3 / Q1.4 显示真实组织有多根 / 双线结构,扩展为多根或加 `treeKey` 维度区分。

#### 3.2.3 候选字段类别(不写字段名)

- 通用字段:`id` / `createdAt` / `updatedAt` / `deletedAt`
- 标识与展示类(节点名 / 简介)
- 父子关系类(自引用外键,可空 = 根节点)
- 节点类别类(指向字典,区分部门 / 小组 / 临时编组)
- 启停状态类(`status`)
- 节点角色类(负责人指向 — 待 §4.1 Q1.11 决定单 / 多 / 不挂)
- 节点扩展属性类(联系方式 / 内部编号 — 待 §4.1 Q1.12 多选决定)
- 临时编组语义类(若 §4.1 Q1.6 选 ① / ②,可能加起止时间字段)

#### 3.2.4 依赖哪些已确认基线

- baseline §1.1 段位 `110xx + 111xx`
- baseline §2.1 通用命名
- baseline §2.2.2.1 organizations vs department 命名澄清
- baseline §2.2.3 启停字段统一 `status`
- baseline §10 软删除显式封装

#### 3.2.5 待调研问题(指向 research.md)

- `research.md §4.1`(对应 questions §3 真实组织结构 14 题):树形拓扑 / 跨树移动 / 节点变更历史 / 节点角色与属性 / V2 写操作支持档位
- 真实组织节点列表**不进**本草案与公共仓库

#### 3.2.6 进入实现前必须确认什么

- §4.1 Q1.1 / Q1.2 / Q1.3:层级深度 / 子组粒度 / 单根 vs 多根
- §4.1 Q1.7 / Q1.8 / Q1.9 / Q1.10:节点变更历史(移动 / 改名 / 撤销 / 合并)— 决定是否引入版本化 / 启停标记
- §4.1 Q1.14:V2 第一阶段写操作支持档位(只读 vs 增删改不可改父级 vs 可拖拽)

#### 3.2.7 设计错了,后续代价是什么

- **迁移成本**:**高**(树形结构调整涉及递归 SQL,数据量虽不大但调试成本高)
- **历史数据修复成本**:中(若启停 / 移动历史未保留,旧业务数据引用的节点语义可能丢失)
- **后续模块耦合成本**:高(members / events 都强依赖组织树形态)

#### 3.2.8 是否建议进入第一阶段

**是**,但**只做最小骨架**(`id` / 标识 / `parentId` / 类别字典 / `status` / 基础时间戳),节点角色 / 扩展属性 / 移动历史等**等 §4.1 调研结论再扩展**。

### 3.3 `members`

> **标签**:[研] / [赖:member_profiles / member_departments / event_participants / attachments] / [敏:否(主表保持克制)] / [史:部分 — 等级 / 部门变更历史待 §4.4 决议] / [先:是(最小骨架)]

#### 3.3.1 解决什么问题

队员主表 — 与 `users`(登录身份)**强制解耦**的组织身份(`research.md §6.1` 红线)。允许 user 没有对应 member、member 没有对应 user、两者 1:1 关联(具体方式见 §4.2)。

**红线**(`research.md §6.2`):**主表保持克制**,只承载"高频读取、低敏感、低变更"字段;扩展资料外移 `member_profiles`;部门归属外移 `member_departments`。

#### 3.3.2 当前倾向

最小主表 + 关联表 + 扩展资料表的"三件套"模式:

- `members` 主表:身份基础 + 启停 + 主部门冗余字段(若 §4.5 选定 `isPrimary`)
- `member_departments` 中间表:多部门关系
- `member_profiles` 扩展表:敏感资料(身份证 / 紧急联系人 / 医疗等,逐字段三问)

**备选方案**:单宽表(把 profiles 字段全合进 members)。
**回退条件**:**禁止回退到单宽表**(违反 `research.md §6.2`)。

#### 3.3.3 候选字段类别(不写字段名)

- 通用字段:`id` / `createdAt` / `updatedAt` / `deletedAt`
- 与 user 关联类(候选 A:可空外键 `userId`;候选 B:无字段,关系走独立关联表 — 见 §4.2)
- 身份标识类(队员业务编号 — 是否有,待调研)
- 称呼展示类(显示名,**不写**真实姓名;在 V2 草案中未必必填)
- 启停状态类(`status`)
- 主部门冗余类(`isPrimary` 在 `member_departments` 还是冗余到 members,待 §4.5 决议)
- 扩展资料指向类(关系到 `member_profiles`)

**禁止进入主表的字段类别**:

- 敏感身份信息类(身份证号等)→ `member_profiles`
- 联系方式类(电话 / 邮箱)→ `member_profiles`(待 §4.3 三问)
- 医疗 / 紧急联系人类 → `member_profiles`(待 §4.3 三问)
- 等级 / 资质类(走字典)→ 关联表或专表(待 §4.2 / §4.5 决议)

#### 3.3.4 依赖哪些已确认基线

- baseline §1.1 段位 `150xx + 151xx`
- baseline §2.1 通用命名
- baseline §2.2.3 启停字段统一 `status`
- baseline §11 v1 接口兼容性(`UserResponseDto` 不倒灌 `members` 字段)
- `research.md §6.1` users / members 解耦红线
- `research.md §6.2` 主表克制原则

#### 3.3.5 待调研问题(指向 research.md)

- `research.md §4.2` 队员等级 / 资质体系(决定等级是否进主表)
- `research.md §4.4` 历史 / 版本化(决定主表变更是否需要历史)
- `research.md §4.5` 一人多部门(决定 isPrimary 字段位置)
- `research.md §5.6` users-members 关联方案(候选 A / B,见 §4.2)

#### 3.3.6 进入实现前必须确认什么

- §4.2 等级 / 资质形态(同维 vs 正交;互斥 vs 多个)
- §4.5 一人多部门常态 / 例外 / 主部门 → 决定中间表 + 主表的字段分配
- §5.6 users-members 关联候选 A / B 选边
- §4.4 哪些字段需要历史(决定是否抽 `members_history` 表 / 仅靠 audit_logs 反推)

#### 3.3.7 设计错了,后续代价是什么

- **迁移成本**:**高**(主表字段集错位涉及全表 reshape,且 member_profiles / member_departments 关联点都要调整)
- **历史数据修复成本**:**高**(若敏感字段误进主表,后续合规整改要做字段下沉 + 数据搬移 + 访问审计补加)
- **后续模块耦合成本**:**极高**(几乎所有未来 V2 业务模块都引用 members.id)

#### 3.3.8 是否建议进入第一阶段

**是**,但**只做最小骨架**(身份基础 + 启停 + 时间戳 + 关联点);扩展字段 / 等级关联 / 主部门冗余等待调研后逐步扩展。

### 3.4 `member_profiles`

> **标签**:[研] / [赖:members] / **[敏:是 — 强制走 research.md §4.3 三问]** / [史:是(变更需保留)[?]] / [先:**待调研**(决定 V2 是否进入)]

#### 3.4.1 解决什么问题

承载队员的**扩展 / 敏感资料**(身份信息 / 联系方式 / 医疗 / 紧急联系人 / 证件附件指向等)。把 members 主表保持克制(`research.md §6.2`)。

**强制约束**(`research.md §4.3` + baseline §8):每个候选字段进入本表前**必须**回答三问:

## 4. 跨模型模式

### 4.1 字典引用方式

#### 4.1.1 候选

- **候选 A**:`<concept>Code` 字符串字段(直接存字典 items.code)
- **候选 B**:`<concept>ItemId` 关联外键(指向字典 items.id)

#### 4.1.2 当前倾向

倾向 **候选 A**(`<concept>Code`),理由:

- 业务侧可读性高(`grade: "A1"` vs `gradeItemId: "cuid-xxx"`)
- 字典 items 的 code 是业务稳定标识(运营看得懂)
- 减少跨表 JOIN(展示时直接拿 code,不必查 items.label)

#### 4.1.3 备选方案与回退条件

- 若 §4.8 决定字典走**双表**(`dict_types + dict_items`)且 items 的 code 在不同 type 下可重复 → 引用必须带 `<concept>Type`(由"概念"语义隐式决定 type),仍可用 code,但字段命名要统一
- 若运营要求"修改 code 后旧引用自动同步" → 改用候选 B(关联外键),代价是 JOIN 增加
- 跨模型统一选边,**禁止**同项目混用

#### 4.1.4 依赖哪些已确认基线

- baseline §2.2.5 字段名与字典关联约定(选定后全模块统一)
- baseline §2.1 命名规范

#### 4.1.5 待调研问题

- §4.8 单表 vs 双表(直接影响 code 是否全局唯一)

### 4.2 users 与 members 关联方式

#### 4.2.1 候选(`research.md §5.6`)

- **候选 A**:`users` 表追加可空外键 `memberId` — 适合"一个登录账号最多绑定一个队员档案"且关联稳定的场景
- **候选 B**:独立关联表 `user_member_links` — 适合"关联本身需要属性(绑定时间 / 绑定人 / 解绑历史)" / "未来可能演化为多对多"

#### 4.2.2 当前倾向

倾向 **候选 A**(可空外键)。理由:

- 关系简单(1:1),用关联表是过度设计
- v1 接口契约不破坏(只在 `users` 加可空字段)
- 查询效率高(无 JOIN)

#### 4.2.3 备选方案与回退条件

- 若 D5 调研发现"账号与队员的绑定关系会变(例如同一队员先后用过多个账号)" → 改候选 B
- 若 D5 调研发现"非队员用户(队委 / IT / 外部协作者)需要登录但永远不绑队员" → 候选 A 仍适用(`memberId` 可空)
- 若 V2.x 演化出"一个账号绑定多个队员档案"诉求 → 候选 A 升级为候选 B,作为独立任务

#### 4.2.4 兼容性红线(`research.md §5.6`)

- **不**删除 / 重命名 / 改类型 v1 `users` 已有字段
- **不**修改 v1 接口的请求 / 响应 DTO 字段集
- **不**在 v1 接口响应中新增"必返"字段(可新增 [可选返回],草案中显式说明)
- **不**把 `members.*` 字段倒灌进 `UserResponseDto`

#### 4.3.1 引用 baseline §10,不重新发明

#### 4.3.2 各表软删策略(本草案级)

### 4.4 审计写入策略

#### 4.4.1 候选

- **候选 A**:**显式** Service 调用 — 每个写操作 Service 方法内 `await this.audit.log({...})`
- **候选 B**:Interceptor / Prisma middleware **自动**捕获写操作

#### 4.4.2 当前倾向

倾向 **候选 A(显式)**。理由:

- 显式 = 可读 = 可审查;符合 "v1 §10 + baseline §10 显式优于隐式" 一脉相承
- Interceptor 难以拿到"业务级语义"(动作 code / 资源 id / 关键字段 before/after);自动捕获到的是"DB 层动作",对人工审计可读性差
- 接入成本可控:每个 Service 方法加一行;漏接不会引发系统故障(只是该操作没审计)

#### 4.4.3 备选方案与回退条件

- 若评审认为"显式接入覆盖率太低,漏接太多" → 启动 D7 后单独立项评估候选 B
- 若 V2 后期发现"读审计需求"(`research.md §4.3 S2`)且要求覆盖所有读操作 → 单独评估 Interceptor

#### 4.4.4 渐进接入纪律

- **不**强制 V2 第一阶段所有写操作都接入审计
- 各业务模块的草案 / 任务卡须**显式**回答"该操作是否需要审计"
- 默认所有"管理员操作他人资料 / 状态变更 / 删除 / 角色变更"必须审计;其他操作按需

### 4.6 events 通用化回退三档(主体讨论,呼应 §3.8)

#### 4.6.1 三档详述

##### 档 (a) 最小 events 骨架

- `events` 表进入,但只含**最小骨架字段**(标题 / 时间 / 类型字典 / 归属组织 / 启停 / 时间戳)
- 各业务场景的差异字段(救援的现场坐标 / 训练的成绩 / 会议的纪要)**不进** events 主表
- 后续每类业务发展时,各自建独立专表(例如 `rescue_operations` / `training_records`),通过 `eventId` 反向关联 events
- `event_participants` 是否进入随后再决定

##### 档 (b) events 进 + 延后 event_participants

- `events` 进入(同档 a 最小骨架)
- `event_participants` **完全延后到 V2.x**
- V2 第一阶段 events 只用于"事件清单查看 / 按组织节点筛选事件"
- 参与人语义在 V2 阶段不承载

##### 档 (c) 完全砍掉

- `events` / `event_participants` **均不进** V2 第一阶段
- 后续每类业务(救援 / 训练 / 会议)各自立项独立表
- 不预先做"通用事件"基础设施

#### 4.6.2 触发回退档的判断

| §4.6 调研结果 | 推荐回退档 |
|---|---|
| 字段差异小(Q6.2 ①)+ 业务对"通用事件清单"有明确诉求(Q6.1 多选)+ 参与状态有清晰需求(Q6.7 ①) | (a) 进入 + 同档 event_participants 进入(本节默认推荐) |
| 字段差异中(Q6.2 ②)+ 参与状态需求模糊 | **(b)** events 进入 / event_participants 延后(**当前默认倾向**) |
| 字段差异大(Q6.2 ③)+ 各业务流程独立性强 | **(c)** 完全砍掉 |
| 调研结果模糊或意见不一致 | 由 D7 评审会拍板,**不**自行选档 |

#### 4.6.3 不论选哪档都不能做的事(硬约束)

- **禁止**为了"通用"强行把救援现场坐标 / 训练成绩 / 会议决议等差异字段塞进 events 主表(`research.md §2.6` 红线)
- **禁止**在 V2 第一阶段为 events 做"完整状态机 / 流程引擎",这是 `research.md §3.6` 暂不做项
- **禁止**在 events 表内承载报名 / 签到 / 请假等业务流程(`research.md §3.16` 考勤业务流程暂不做)

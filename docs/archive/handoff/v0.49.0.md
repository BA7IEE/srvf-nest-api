## 2. 行为矩阵与边界

- `team-leader@root`:全队队员轴与 participation 可读可管;敏感明文、考勤终审、Recruitment/team-join 仍按既有独立授权。
- `dept-leader@non-root`:本部门及子树可读可管;跨部门列表不出现,detail/write `30100`。
- `vice-captain` / `dept-deputy` / `deputy-group-leader`:对应范围只读;所有写 action 恒 `30100`;`org-readonly` 10 码、`group-readonly` 11 码均动态投影且零 `*.read.sensitive`。
- `group-leader`:沿 `group-manager` 既有码集管理本组,本版未增删正职角色码集。
- supervision holder:沿 `org-supervisor` 4 个既有码只读所分管组织;本版只接消费侧,角色码集不变。
- GLOBAL `biz-admin` / SUPER_ADMIN:全队行为保持不变。Recruitment/team-join 继续中央流程;users/content/notifications/audit-logs/App API 不在本版可见性改造范围。
- 冻结边界机械核验:`src/modules/recruitment/**`、`src/modules/team-join/**`、`src/modules/auth/**`、`src/modules/permissions/rbac.service.ts`、Prisma schema/migrations、BizCode、AuditLogEvent 相对 `v0.48.0` 全部 0 diff。

## 4. 部署与前端联调

- **数据库**:本版 0 migration,仍先执行常规 `prisma migrate deploy` 并确认 no pending。**必须幂等运行一次 `pnpm prisma:seed`**以创建/同步 `org-readonly`、`group-readonly` 与三条副职 policy;禁止手工 SQL 复制码集或做存量数据订正。
- **运维重配**:上线后按 [`scoped-authz-go-live-checklist.md §6.2`](../../ops/scoped-authz-go-live-checklist.md)逐人执行:补 active 任职 → 验 effective-permissions/authz explain → 收回其临时 `biz-admin` GLOBAL 绑定 → 抽查行为矩阵。Recruitment/team-join 审核人保留明确的中央流程授权。
- **后台 FE**:菜单仍前端静态维护,改用 `authz/me/effective-permissions` 的 `permissions[]` 过滤;不要新增菜单树 API。读码只控制展示候选,写按钮必须按真实 create/update/delete/approve action 码过滤,后端仍会结合 resource ref/scope 二次判定。
- **列表**:members/options 与 registrations/attendance-sheets 的 `organizationId`/`includeDescendants` 仅是用户筛选,服务端会与 Authz scope 求交;前端不应把 query 参数当授权来源。member/certificate 范围只认 active PRIMARY membership。
- **敏感字段**:副职和分管角色不持 `*.read.sensitive`;继续沿既有掩码/null 回写防护,不得把掩码值或 null 覆盖真实资料。
- 不回改本 handoff;发布后事实由 `docs/current-state.md §1/§2/§4` 与 `docs/handoff/openapi.json` 承接。

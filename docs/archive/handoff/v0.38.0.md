## 1. TL;DR

- **v0.38.0 = 队员账号闭环(v1 MVP + v2 完整生命周期)** — 给"已存在但没有登录账号"的队员(手动建档 / 历史迁移队员)补齐"从零开号"到"绑定 / 解绑 / 退号重开 / 启停 / 批量开号"的完整账号生命周期管理能力。v1 立项 = goal「队员账号闭环 v1(MVP)」(无冻结评审稿,C 档);v2 立项 = goal「队员账号闭环 v2(完整生命周期)」,冻结评审稿 [`member-account-loop-v2-review.md`](../reviews/member-account-loop-v2-review.md)。**这是该功能线的首次发版**(v1 从未单独发版,累计在 `## Unreleased` 直到本次一并折叠),故无"对已发版行为的破坏"可言。
- **6 笔 CHANGELOG 条目 + 1 笔纯 docs T0(共 7 PR,#515–#521)**:
  - **v1 MVP 开号**(#515,C 档):新 `POST admin/v1/members/:id/account`(body `{phone}`)——随机不可用密码 + `username=memberNo` + `phone`/`phoneVerifiedAt` 管理员背书,新码 `member.grant.account` 绑 ops-admin;队员面 additive `hasAccount`/`accountStatus`/`userId` + `?hasAccount=` 过滤;用户面 additive 可选 `memberId`/`member` 回显(仅 admin list/detail)。开号后走既有 `login-sms` 登录,`src/modules/auth/**` 零改动。
  - **P2002 并发补丁**(#516,B 档):`grantAccount` 并发开号在 `username`/`memberId` 双唯一约束下可能裸 500,补 `memberId` 分支映射为既有 409 `MEMBER_HAS_LINKED_USER`。
  - **T0 评审稿冻结**(#517,docs-only,不进 CHANGELOG):v2 完整生命周期立项冻结。
  - **v2 Schema 刀**(#518,D 档,第 40 migration):`User.memberId` 全量 `@unique` → partial unique `WHERE deletedAt IS NULL`(软删旧号释放槽位);连带 `Member.user`→`users` 一对多改造 + 2 消费方(`authz/resource-resolver.service.ts` / `notifications/birthday-greeting.service.ts`)同步;零 API 可见行为变化。
  - **v2 Endpoint 刀**(#519,D 档,+4 端点):`bind`(认领悬空账号)/ `unbind`(只断链)/ `reopen`(软删旧号+开新号原子,`username` 代际后缀化解冲突)/ `PATCH .../status`(队员面启停,复用 `user.update.status`,联动撤销 refresh token)。新码 `member.bind.account` 绑 ops-admin;+2 BizCode(`15032`/`15033`)+ 3 audit event。**D-2 唯一有意行为变更**:软删旧号后可重新开号。
  - **v2 批量刀**(#520,D 档,+1 端点):`POST members/accounts/bulk-grant`(1-200 条,逐行 skip-on-error,每行独立事务,0 新码)。
  - **computeNextUsername 探测式补丁**(#521,B 档,元核验发现边角):`unbindAccount` 只断链不软删导致悬空账号仍占用 username,旧按 `count` 推算代际的算法在"先解绑再开号"场景 100% 撞号;改为直接探测首个未占用 username(含悬空账号),不再依赖 `memberId` 参数。与维护者当场拍板的行为变更:username 冲突不再区分"自己的历史占用"与"巧合的他人占用",两者在 DB 层本就无法区分。

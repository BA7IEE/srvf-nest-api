## 2. 元核验与缺陷修复(本版重点)

主会话对 #630 做五维元核验(足迹 / 递补引擎 / 状态机与 create 分流 / e2e 矩阵 / 禁区与零 diff)+ 对抗性复核:22 条候选发现经"默认推翻"复核后存活 6 条,其中 **1 条真实并发缺陷已修(#631)**。

- **缺陷**:`activities.service.ts` 的 capacity update 中,delta 基线 `current.capacity` 取自 `FOR UPDATE` **之前** 55 行的无锁读,而 `claimAtStatus` 只 CAS `statusCode`——capacity 的 read-modify-write 无保护。并发/重试的同值扩容会按陈旧基线各算一次 delta(capacity 1→3、双击两次:各算 delta=2 → 递补 4 名,净增名额仅 2);先 5→10 提交后携 `capacity=6` 的**缩容**请求会用陈旧基线 5 判定 `6>5` 成立而递补,违反 W3。
- **影响面**:容量从未被击穿(approve 容量闸始终兜底),实际危害 = 多余递补 + 向这些成员误发「候补已递补」通知 + 他们 approve 时必被 `ACTIVITY_CAPACITY_EXCEEDED` 拒。
- **修法**:delta 基线改为取锁后重读,与同为取锁后读的 `passCount` 对齐同一快照。补并发回归 e2e(**变异验证**:还原旧逻辑该用例立刻红、其余 185 tests 仍绿,证实此缺陷此前零测试覆盖)+ 递补通知 `notificationTypeCode`/`channels` 断言补强。
- **递补引擎其余部分核验通过**:锁序全仓一致 Activity→Registration(无死锁环)、FIFO 正确、CAS 不死循环、通知 commit 后事务外、audit 复用 `registration.review` + `extra.action='promote'`(event 名未动)、并发双取消不重复/不漏递补。
- **文档追认**:`docs/participation-bounded-context.md` 与 `src/modules/activity-registrations/CLAUDE.md` 的状态机 4 态→5 态 true-up 经维护者事后追认(实现必需,不改则文档失真;原 goal 授权清单遗漏该项,后续 goal 模板已将「实现必需的文档 true-up」默认列入授权)。

## 3. 部署与前端联调

- **数据库**:0 migration(仍 52),`prisma migrate deploy` 应报 no pending。**必须幂等运行一次 `pnpm prisma:seed`** 以创建 `registration_status.waitlisted` 字典项 —— 缺 seed 则候补态无法通过字典校验。
- **环境变量 / cron**:零新增,运维零动作。
- **⚠️ miniapp 必须适配**:①满员报名现在返回 **200 + statusCode=waitlisted**(不再是 21031 错误),不适配会把成功当失败;②报名列表/详情新增 `waitlistPosition`(候补从 1 起),应展示「第 N 位候补」;③候补者可自行取消(cancelMy 已接受 waitlisted);④递补时收到 `registration-result` 类型站内通知「候补已递补」;⑤候补者不能签到(打卡只认 pass),按钮态需据此判定。详见 [`docs/handoff/miniapp.md`](../../handoff/miniapp.md)。
- **admin-web**:候补列表 = 既有报名列表按 `statusCode=waitlisted` 过滤(无新端点);递补是**自动**的——取消 pass 或调大 capacity 后队首自动进待审,无需手动点;dashboard 新增 `registrations.waitlisted` 计数可做提示位;approve 仍需人工(候补不直通 pass)。详见 [`docs/handoff/admin-web.md`](../../handoff/admin-web.md)。
- 不回改本 handoff;发布后事实由 `docs/current-state.md §1/§2` 与 `docs/handoff/openapi.json` 承接。

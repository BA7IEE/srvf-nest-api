## 7. 工作节奏原则

### 7.1 可提速(低风险,沿本 PR 范式)

- **低风险文档**:handoff 增量 / CHANGELOG Unreleased 修正 / 过期措辞清理 / 业务访谈纪要起草
- **重复迁移类**:若 C-2 read 类一旦由 E-1 业务确认稿启动,沿 PR #3 / #4 / #5 / #6 已验证的 "单模块独立 PR + 单 docs 收口 PR" 范式继续推进(零新概念)
- **手工验收记录**:Swagger UI / `/api/docs` 试调

### 7.2 必须减速(高风险,先评审后实施)

- **schema / migration**:任何 Prisma schema 写入(尤其敏感字段"先占位")
- **权限模型**:扩 `Role` enum / 引入 RBAC / 修改 Guard 顺序 / 改 `JwtStrategy.validate()` 查库逻辑
- **敏感字段**:身份证 / 紧急联系人 / 医疗信息 / 证件附件 — 业务用途 / 查看角色 / 保存期限三问未答完,**不进 schema**
- **业务边界不清**:统计口径 / dryRun 语义 / 批量重算策略 — 业务方未拍板,**不开发**
- **基础设施扩张**:Redis / 队列 / 定时任务 / Provider 实装 — 沿 V1.1 §11.3 仍锁定

判定提速 / 减速速查:

- 仅文档 / 已评审稿内代码迁移 / 既有范式延伸 → **可提速**
- 涉及 schema / migration / 权限模型 / 敏感字段 / 业务规则定义 / 基础设施扩张 → **必须减速**

减速不是不做,而是必须先评审。从"快速实现"切换到"想清楚再写"的判定准绳:**风险是否可逆 + 业务是否拍板 + 红线是否触碰**。

## 4. 当前能力(v0.9.0 落地后)

### 4.1 4 张 RBAC 表(沿 D7 v1.1 §4)

| Prisma model | DB 表名(`@@map`) | 软删 | 唯一约束 |
|---|---|---|---|
| `RbacRole` | `roles` | 软删(`deletedAt`)| `code` |
| `Permission` | `permissions` | 物理删 | `code` |
| `RolePermission` | `role_permissions` | 物理删 | `(roleId, permissionId)` |
| `UserRole` | `user_roles` | 物理删 | `(userId, roleId)` |

**v1 enum `Role` 保持不动**(`SUPER_ADMIN / ADMIN / USER` 三层永远不变;沿 D7 v1.1 命名修订 B1 + A-4 红线);RBAC 4 表与三层 Role 并存(沿 D12 永不切换)。

### 4.2 16 个 RBAC 端点(沿 D7 v1.1 §5.1 F2)

| 端点 | 方法 | 入口 Guard |
|---|---|---|
| `/api/v2/permissions` 列表/创建/更新/删除 | GET/POST/PATCH/DELETE × 4 | `@Roles(SUPER_ADMIN, ADMIN)` |
| `/api/v2/roles` 列表/详情/创建/更新/软删 | × 5 | `@Roles(SUPER_ADMIN, ADMIN)` |
| `/api/v2/roles/:id/permissions[/:permissionId]` 批量授权/撤权 | POST/DELETE × 2 | `@Roles(SUPER_ADMIN, ADMIN)` |
| `/api/v2/users/:userId/roles[/:roleId]` 查/分配/撤销 | GET/POST/DELETE × 3 | `@Roles(SUPER_ADMIN, ADMIN)` + Q7 C2 中庸 |
| `/api/v2/rbac/me/permissions` | GET × 1 | `@Roles(USER, ADMIN, SUPER_ADMIN)` |
| `/api/v2/rbac/reload` | POST × 1 | `@Roles(SUPER_ADMIN, ADMIN)` |

**入口仍走 `@Roles(...)`**;`rbac.can()` 在业务模块的实际调用**未接入**(本批次 0 处业务调用)。

### 4.3 BizCode 段位 `300xx + 301xx`(14 项)

- `300xx` 通用(10 项):`PERMISSION_NOT_FOUND` / `PERMISSION_CODE_ALREADY_EXISTS` / `INVALID_PERMISSION_CODE_FORMAT` / `ROLE_NOT_FOUND` / `ROLE_CODE_ALREADY_EXISTS` / `ROLE_DELETED` / `INVALID_ROLE_CODE_FORMAT` / `ROLE_PERMISSION_NOT_FOUND` / `USER_ROLE_ALREADY_EXISTS` / `USER_ROLE_NOT_FOUND`
- `301xx` 权限/边界(3 项):`RBAC_FORBIDDEN`(段位预留,本期未在业务模块调用)/ `LAST_OPS_ADMIN_PROTECTED` / `CANNOT_ASSIGN_HIGHER_ROLE`

### 4.4 RbacService 判权核心

- `getUserPermissionCodes(userId)` → `Set<string>`:走 `RbacCacheService` 缓存
- `can(user, action, resource?)` → `boolean`:SUPER_ADMIN 短路 → 缓存查询 → 精确匹配 → `.self` ownership(沿 D7 §8.2)
- `judge(user, action, resource?)` → `RbacJudgeResult`:同 `can()` 但带 reason
- `getMyPermissions(user)` → `MyPermissionsResponseDto`:`me/permissions` 端点入口
- `reload({ scope })` → `ReloadRbacResponseDto`:`reload` 端点入口(3 档 scope all/user/role)
- `checkOwnership(user, resource)`:`.self` 路径用 `user.id` / `user.memberId` 混合判定(沿 D7 §8.3)

### 4.5 RbacCacheService 进程内 TTL 缓存

- 实现:Map + setTimeout 等价;**无新依赖**
- TTL:`RBAC_CACHE_TTL_SECONDS` env(默认 1800 秒 / 30 分钟;推荐区间 [60, 86400])
- 3 个 invalidate 入口:`invalidateUser(userId)` / `invalidateAllUsersWithRole(roleId)` async / `invalidateAll()`
- 缓存粒度:`user:<userId>` → `Set<string>` 权限 code 集

### 4.6 seed/bootstrap

- `seedRbac()` 函数:14 条 `rbac.*` Permission upsert + `ops-admin` RbacRole upsert + 14 条 RolePermission 映射
- bootstrap:`RBAC_INITIAL_OPS_ADMIN_USER_ID` env 优先 → SUPER_ADMIN fallback
- 强校验:seed 完成后至少 1 个活跃 `user_role` 持有 `ops-admin`,否则 throw 退出
- 全部幂等(upsert + `update: {}`;重复跑零增量)

### 4.7 CurrentUserPayload 扩展

- `+ memberId: string | null`(沿 D7 §8.3 owner 判定)
- `JwtStrategy.validate()` select 同步追加
- **v1 14 接口 response 契约 zero drift**(memberId **不**进 `UserResponseDto` / `userSafeSelect`)

### 4.8 测试覆盖

- 7 个 e2e spec:`permissions` / `rbac-roles` / `role-permissions` / `user-roles` / `rbac-me-permissions` / `rbac-reload` / `seed-rbac`

## 9. 已知 quirk:`gh pr merge` worktree checkout main 冲突预案

本仓库在 `.claude/worktrees/` 下创建临时 worktree 时,`gh pr merge --squash --delete-branch` 会撞:

```
failed to run git: fatal: 'main' is already used by worktree at '/Users/.../srvf-nest-api'
```

**根因**:主仓 worktree 占用 `main` 分支,gh 试图本地 checkout main 失败。

**预案**(沿 PR #54-#63 累计 10 次验证成功):

```bash
# 1. 仍执行 gh pr merge(GitHub 端会完成 squash merge)
gh pr merge <PR#> --squash --delete-branch
# → 报 worktree 错误,但 GitHub 端已 MERGED

# 2. 验证 GitHub 端 MERGED 状态
gh pr view <PR#> --json state,mergedAt,mergeCommit
```

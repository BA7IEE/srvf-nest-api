## 一、背景与硬边界

### 1.1 这次访谈解决什么

队里管理系统计划上线**附件功能**——让系统能挂"图片 / 证件照 / PDF / 资料文档"等文件。功能上线后,前端能上传、能展示、能下载。

附件功能不是简单"上传一张图",涉及:

- **挂在哪个业务对象上**(队员档案?证件?活动?装备?)
- **谁能上传 / 谁能查看**(本人 / 管理员 / 部长)
- **能保存多久 / 离队是否清理**(合规问题)
- **存在哪里**(本地服务器 / 阿里云 OSS / Cloudflare R2 等)

后两条决定数据合规风险与运营成本,**必须在上线前由业务方明确**;前两条决定数据模型走向,**做错了后续要做 migration,有成本**。

### 1.2 不在本次访谈范围内的事

| 项 | 留给谁 |
|---|---|
| 用什么云存储(本地 / OSS / R2 / S3 / MinIO) | D7 评审 + Provider 选型安全策略评估 |
| 数据库表结构 / 字段名 / 索引 / FK / migration | D7 attachments 评审稿 |
| API 路径 / 入参 / 出参 / BizCode 详码 | D7 评审稿(段位 `130xx` / `131xx` 已锁) |
| 客户端上传方式(直传 vs 中转 + 签名 URL) | D7 选型 |
| 与 APD 部长 / 副部长权限的对接 | C-6 RBAC 模型评审独立走 |
| 字典 seed 真实值(部门 / 等级 / 活动类型) | 用户私下提供,不进公共仓库历史 |

### 1.3 一条事先告知的边界

> **附件归属的业务对象(挂在谁身上)决定第一版的数据模型;一旦上线后想改"挂载对象",已经上传的文件元数据迁移成本中等。所以在第一批上线前确认"首发场景挂在哪里"很重要。**

这意味着:**第一批上线时定下的"首发挂载对象"是数据模型的基础**。如果上线后才发现归属错配,**已有附件需要走数据迁移脚本**——可做,但比一开始就定准多一道工作量。

### 问题 2:附件元数据**除了基础字段,还需要哪些**

研究文档 §2.4 列出 attachments 必有的基础字段(段头标 **[当前倾向]**,本期仍属于待业务方确认范围):

- `key`(文件标识,Provider 侧的唯一引用)
- 原始文件名(`originalName`)
- MIME 类型
- 文件大小
- 上传者(`uploadedBy` → User.id 或 Member.id,二选一在 D7 决议)
- 上传时间(`uploadedAt`)
- 归属业务对象(`ownerType + ownerId` 或专表 FK,见问题 4)

本题问的是**额外字段是否要**:

| 候选额外字段 | 用途 | 是否要 |
|---|---|---|
| `checksum`(SHA-256 哈希) | 防重复上传 + 完整性校验 + 秒传基础 | 待业务确认 |
| `etag` | S3 兼容 Provider 的服务端版本号 | 待业务确认 |
| `description`(用户备注) | 上传时给附件加文字说明(如"2024 年体检报告") | 待业务确认 |
| `accessLevel`(public / internal / sensitive) | 决定查看权限分级 | 待业务确认(关联问题 7) |
| `tags`(标签数组) | 让附件按标签筛选(如"已审核""待复核") | 待业务确认 |
| `originalUploaderName`(冗余存上传者人名) | 上传者离队 / 软删后仍能看到人名 | 待业务确认(关联软删策略) |
| `expireAt`(附件过期时间) | 让附件本身有"有效期"(如年度体检报告) | 待业务确认 |

**[当前 AI 参考观察]**:research §2.4 锁定"仅元数据,不实装 Provider";额外字段集 §2.4 写"留待 D7"。**业务方层面只需告知"哪几个字段在业务流程里真的会用到"**;technical 字段(checksum / etag)由 D7 评审拍。

**不确认的影响**:D7 阶段会按"必备基础字段"硬走,额外字段缺一进 schema 后要走 migration,变更成本中-低。

### 问题 3:**一个附件能挂在几个业务对象上**

| 候选 | 含义 |
|---|---|
| A | **单归属 1:N**(一个附件只属于一个对象;同一张图给两个对象就上传两次) |
| B | **多对多 M:N**(一个附件可以挂多个对象,共享同一份物理文件) |

**[当前 AI 参考观察]**:research §4.9 默认 **[当前倾向]** 单归属 1:N;理由是业务首期未出现常态多归属诉求,且单归属逻辑简单、风险低。本期需要业务方确认。

**不确认的影响**:

- 选 A 后续真出现常态多归属诉求,需要加关联表 migration(成本中等)
- 选 B 早期就要建关联表,若实际多归属诉求是冷门,等于过度设计;且**多对多场景下删除策略复杂**(其他对象还在引用时,主对象删除附件能不能删?物理文件能不能清?)

### 问题 4:附件归属用"**多态外键**"还是"**每业务表自己挂 attachments**"

| 候选 | 含义 |
|---|---|
| A | **多态外键**(`ownerType` + `ownerId`):一张 `attachments` 表通吃所有业务模型;新增业务模型时不需要动 attachments 模块 |
| B | **每业务表自己挂**:每种业务模型独立挂自己的附件表(如 `member_attachments` / `certificate_attachments` / `activity_attachments`),通过 FK 关联 |
| C | **混合**(部分模型走 A,部分走 B) | 

**[当前 AI 参考观察]**:research §4.9 标 **[待调研]**;A 灵活但失去数据库层的 FK 完整性(`ownerType` 是字符串字段,数据库不约束)、B 安全但表数量随业务模型膨胀。**业务方层面只需告知"未来想新增一类业务(如装备 / 训练科目),能不能不动 attachments 模块就挂上"**;C 是技术折中,本期不强求业务方拍。

**不确认的影响**:

- 选 A 后续每加业务模型,attachments 不动;但需要在 Service 层手写校验 `ownerType` 合法性
- 选 B 加业务模型要新建中间表(成本中等),但 FK 完整性强,数据库层不会出现"orphan attachment"
- 选 C 留待 D7 拍

### 问题 5:附件**什么时候算"删除"**

| 候选 | 含义 |
|---|---|
| A | **软删除**——按下"删除"按钮后元数据保留(标记 `deletedAt`),物理文件保留 N 天后清理(N 由业务方定) |
| B | **硬删除**——按下"删除"后元数据立即从库里消失,物理文件同步删除(不可恢复) |
| C | **跟随归属对象**——业务对象软删时附件元数据也软删;业务对象硬删时附件也硬删 |

**[当前 AI 参考观察]**:research §5.2 **[当前倾向]** 附件元数据软删除;§6.5 写"附件元数据软删除待草案"。

**不确认的影响**:

- 选 A:误删可救回(N 天内),物理文件占用增加(N 天延迟清理)
- 选 B:误删不可救回;物理存储占用最优
- 选 C:跟随业务对象生命周期,逻辑一致;但**业务对象的软删 vs 硬删策略本身就分场景**(如 audit_logs 不可删 / Member 软删 / 字典 items 待定),C 选项需要业务方逐场景澄清

### 问题 6:**谁能上传**

| 候选 | 含义 |
|---|---|
| A | **全员**(SUPER_ADMIN / ADMIN / USER 队员本人) |
| B | **限管理员**(SUPER_ADMIN / ADMIN);队员不能上传 |
| C | **按附件类型分**(如证件照只允许队员本人上传,活动照允许管理员上传) |

**不确认的影响**:默认按 B 上线(最严);后期想放开队员上传需要在 Service 层加判定,**不动 schema**。

### 问题 8:**单个文件最大尺寸 / 允许的 MIME / 是否做病毒扫描**

#### Q8a 单个文件最大尺寸

| 候选 | 含义 |
|---|---|
| A | 1 MB(只够小照片 / 文本) |
| B | 5 MB(常规照片) |
| C | 10 MB(高清照片 / 短 PDF) |
| D | 50 MB(长 PDF / 资料汇编) |
| E | 100 MB+(视频 / 大文件) |
| F | 按附件类型分级(证件照 5MB / 活动照 10MB / 资料 50MB) |

#### Q8b 允许的 MIME

候选(可多选;**不限于此**):

| MIME | 含义 |
|---|---|
| `image/jpeg` / `image/png` / `image/heic` / `image/webp` | 图片 |
| `application/pdf` | PDF |
| `application/vnd.openxmlformats-officedocument.wordprocessingml.document`(docx) / `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`(xlsx) | Office 文档 |
| `text/plain` | 纯文本 |
| `video/mp4` / `video/quicktime` | 视频(如要) |
| `application/zip` / `application/x-rar-compressed` | 压缩包(**安全风险较高**) |

#### Q8c 是否做病毒扫描

| 候选 | 含义 |
|---|---|
| A | **不做**(默认信任上传者) |
| B | **上传时同步扫描**(影响上传响应时间) |
| C | **异步扫描**(上传成功后后台扫描,可能事后标记可疑) |

**[当前 AI 参考观察]**:

- research 未涉及此层

**不确认的影响**:默认按"无病毒扫描 + 标准 MIME 白名单";恶意文件上传风险由业务方知情承担。

### 问题 9:保存期限 / 合规 / 加密——按场景三问(research §4.3 延伸)

| 候选场景 | Q1:队员离队后是否自动清理 / 标记 | Q2:法务保存期限要求 | Q3:是否要求加密存储 |
|---|---|---|---|
| 队员证件照 | ? | ? | ? |
| 活动现场照 | ? | ? | ? |
| 培训资料 | ? | ? | ? |
| 装备图 | ? | ? | ? |

**为什么必须三问**:research §4.3 + §5.6 锁死敏感字段必须有明确的合规依据;**离队后的附件处理 = 数据最小化原则的硬要求**;**加密存储 = 是否启用 KMS / SSE 的决策依据**。

**[当前 AI 参考观察]**:

- "永久保存"是默认做法,但**对敏感信息默认永久保存可能违反《个人信息保护法》最小化原则**——业务方需要给法务依据
- "离队后自动清理"涉及**触发条件**(队员 status 改 DISABLED?Member 软删 deletedAt?)、**清理动作**(只标记不删 / 物理删 / 仅元数据删保留物理文件)、**保留期限**(清理后还存几天?)三个子问题
- "加密存储"实施成本中等(Provider 侧 SSE 通常开箱即用);但**KMS 密钥管理**需要业务方决定密钥归属(自管 / 委托 Provider)

**不确认的影响**:**未三问的场景不进 schema**(V2 §18.4 硬约束);离队后附件处理涉及个保法合规风险,业务方需书面确认。

### 问题 10:现有 `Certificate.attachmentKey` 占位字段**如何处理**

现有事实:

- `prisma/schema.prisma` 的 `Certificate` 模型包含 `attachmentKey` 字段(可选字符串,最大 256 字符;批次 6a 协议占位)
- 该字段是 v0.3.0(2026-05-10)批次 2 落地时留下的协议占位,**当前始终 NULL**(数据库层未启用)
- attachments 模块实装后必须明确这个字段的去向

| 候选 | 含义 |
|---|---|
| A | **升级为指向 attachments 的引用**:`Certificate.attachmentKey` 改为存 `attachments.key`(或加 FK 到 `attachments.id`),保持现有占位的语义连续性 |
| B | **废弃此字段**:`Certificate` 与 attachments 通过"通用归属关系"(问题 4 选 A 多态外键 或 选 B 中间表)关联,`Certificate.attachmentKey` 字段删除 |
| C | **两者并存**(不推荐,留下迁移债务) |

**[当前 AI 参考观察]**:

- v0.8.0 handoff §6.1 锁"不动 v1 14 接口";`Certificate` 模型属于 V2 批次 2,不是 v1 14 接口,**理论上允许动**但需 D7 评审
- 选 A 实施最简(字段含义平滑升级,无 migration 改字段);选 B 干净但需要 migration 去字段
- 选 B 与问题 4 选 A(多态外键)的整体设计更一致

**不确认的影响**:选 A 最快上线,但与"通用 attachments"设计有耦合;选 B 干净但需 schema migration。

### 问题 11:与 `member_profiles` / `events` 复活的耦合度

事实背景:

- research §4.9 列出 attachments 的挂载对象包括"队员档案 + 活动 / 事件"
- `member_profiles`(队员档案扩展)和 `events`(事件)目前都在 V2 第一阶段**延后**状态(v0.8.0 handoff §5.3 Slow-3 / Slow-4)
- attachments 复活的触发条件之一就是"`member_profiles` / `events` 解锁"(V2 红线 C-7)

| 候选 | 含义 |
|---|---|
| A | **三者独立**:attachments 先行,覆盖现有模型(`Member` / `Certificate` / `Activity`),等 `member_profiles` / `events` 复活时再扩 |
| B | **三者捆绑**:attachments 必须等 `member_profiles` / `events` 一起复活,避免后续重做 |
| C | **部分耦合**:attachments 与 `events` 一起做(如果 events 是首发场景),`member_profiles` 单独走 |

**[当前 AI 参考观察]**:

- research §4.9 列出候选未拍板
- A 风险:后续 `member_profiles` 复活时可能要扩 attachments 字段(如加 `accessLevel` / `expireAt`);但同时 attachments 本期上线时间更早,业务方更早能用上
- B 风险:attachments 启动被合规依据未补齐(`member_profiles` 复活硬前置)、用户拍板未到位(`events` 复活硬前置)的两个延后模型拖慢;时间不可控
- C 风险:折中方案,需要业务方明确"events 的真实业务需求"(handoff §5.2 E-14 标"现有 activities/certificates/attendances 已承载大部分场景")

**不确认的影响**:默认按 A;但若问题 1(首发场景)选 B(活动现场照),则 A 与"挂在 events 上"的事实冲突,**回头需要重新对齐**。

## 1. 背景与动机

### 1.1 业务画像

深圳公益救援队是一支以志愿者为主体的民间应急救援组织,内部系统的潜在使用者包含:

- 队委会与日常管理人员(管理后台)
- 各部门负责人(部门内部成员管理)
- 普通队员(本人资料维护、活动报名 — 形态待调研)

业务范围**潜在涉及**(均**非本阶段实现目标**,仅供画像参考):队员档案、组织结构、活动记录、训练记录、装备登记、出勤、证件/资质、对内公示。

### 1.2 为什么要先做"基础数据底座",而不是直接做救援业务

**直接做业务的风险**:

- R-A 救援业务流程未稳定:深圳救援队的部门划分、等级体系、活动类型在不同年度、不同业务条线可能不同,直接落表会导致后续大改
- R-B 业务模块之间高度耦合于人员与组织数据:训练、出勤、装备、活动如果各自独立建模,会重复出现"队员是谁""属于哪个部门""哪一年加入"的字段
- R-C AI 辅助开发的最大短板是"长期一致性":如果第一批业务表各自定义"队员",未来 AI 改一个业务模块时极易破坏另一个的隐式契约

**先做基础数据底座的收益**:

- 把"人员主数据 / 组织 / 字典 / 附件 / 审计"这五类**业务模块共用**的能力沉淀下来
- 把"哪些枚举可变(走字典)、哪些枚举固化(走代码)"这条边界提前划清
- 让后续每个业务模块只专注其特有的字段与流程,共用部分零重复

### 1.3 与 v1 / V1.1 / 模板 freeze 的关系

- **v1**(`ARCHITECTURE.md` §1-§10):auth + users + 统一异常 / 响应 / Swagger / Prisma 软删除约定 — **全部保留生效**
- **V1.1**(`ARCHITECTURE.md` §11):pino 日志 / helmet / throttler / terminus 健康检查 — **全部保留生效**
- **模板 freeze**(README v0.1.6):main 分支仅 docs / CI 触发路径变更 — 本派生项目的工作**不回流**到模板仓
- **V2**:本研究文档 + 未来 `srvf-foundation-data-model-draft.md` + 未来 `ARCHITECTURE.md §12` 共同界定其范围,**仅对 srvf-nest-api 生效**

## 2. 第一阶段要解决什么(范围正向声明)

> 本节列"问题域",不列"解法"。每条只描述"要解决的问题",不预设"用什么表/字段/接口解决"。

### 2.1 [已确认] 人员主数据:登录身份与组织身份解耦

**问题**:v1 的 `users` 表只承载"能登录系统的账号",但救援队内部存在两类身份的天然分离:

- **登录身份(user)**:能进入系统的账号 — 可能是队委、部门管理员、IT 维护者,也可能不是队员
- **组织身份(member)**:队员档案 — 不是所有队员都需要登录系统,也不是所有登录用户都是队员(例如外部协作者、IT 顾问)

两者必须**解耦建模**,允许:user 没有对应 member、member 没有对应 user、user 与 member 1:1 关联。

### 2.2 [当前倾向] 组织结构:树形,但不预设具体部门

**问题**:救援队存在层级结构(队 → 部门 → 小组?),但**实际层级深度与命名 [待调研]**。

设计层面要解决的是"提供组织树这一形态",而**不是**预先内置"水域救援队""山地救援队"等具体节点。所有节点由后续运营人员或 seed 在评审后录入。

### 2.3 [当前倾向] 字典:把"可变枚举"从代码中迁出

**问题**:救援队的"队员等级""活动类型""证书类型""装备类别"等取值会随业务调整。如果硬编码为 Prisma enum 或 TS 联合类型,每次调整都要改 schema + migration + 前端,改动成本高。

设计层面要提供"字典基建",但**不预先定义任何字典内容**。具体字典类型与取值在评审后由 seed/运营录入。

**红线**:系统级、跨业务、稳定不变的枚举(如 v1 的 `Role` / `UserStatus`)**不入字典**,继续走 Prisma enum。

### 2.4 [当前倾向] 通用附件元数据

**问题**:多个未来业务模块(队员证件、活动现场照片、装备照片、培训资料)都需要"挂附件"的能力。如果每个模块各自落字段,会出现 N 套相似但不一致的元数据结构。

设计层面要沉淀"附件元数据这一通用结构",**仅元数据**:文件标识(`key`)、原始名、MIME、大小、上传者、归属业务对象等。**不实装** Provider(本地/OSS/R2),沿用 v1 §1 的 `common/storage/` 仅 interface 不实现的模式。

### 2.5 [当前倾向] 审计日志的"基础设施"

**问题**:救援队的人员变动、部门调整、敏感信息访问都属于"事后需要追溯"的事件。完全靠 Git 提交记录或日志文件不够。

设计层面要提供"审计日志这一表 + 写入入口",但:

- **不预先决定**哪些操作必须写审计(写入策略 [待调研])
- **不**强制每个 service 立即接入(允许渐进式接入)
- **不**承诺"全量操作审计"(那是合规级要求,不是 V2 范围)

### 2.6 [当前倾向] 通用事件 / 参与关系作为"候选底座模型"

**问题**:救援队存在大量"按时间发生、有参与人、有附件"的活动:救援出动、训练、会议、公益活动、考核。如果每类都建一张专表,会出现多张结构相似的表。

设计层面**研究**(不是实现)是否能用"通用事件 + 参与关系"作为底座,让具体业务类型靠"事件类型字典"区分。**这是研究目标,不是已确认的实现方案**,详见 §4.6。

**通用化失败时的回退策略(硬约束)**:如果 §4.6 调研结果显示救援任务、训练、会议、公益活动之间的字段差异过大,**禁止**为了"通用"强行往 events 主表塞各场景特有字段把它做成大宽表。允许的回退路径:

- (a) V2 仅保留**最小 events 骨架**(标题、时间、类型、负责人、归属组织等无争议字段),具体场景所需的差异字段由后续业务模块各自承载
- (b) V2 完全延后 `event_participants`,只保留 `events` 骨架
- (c) V2 完全砍掉 events / event_participants 两张表,等具体业务模块立项时再决定建模

具体回退到哪一档,由 §4.6 调研结果与草案评审决定。

## 3. 第一阶段暂不解决什么(显式不做清单)

> 本节是**硬性排除清单**,凡列入此节的方向,即使后续看起来"顺手就能加",也禁止在 V2 第一阶段实现。
> 标 [暂不做] 的项**不等于永远不做**;用户对该项的态度可在未来里程碑重新评估。
> 部分条目附"为什么"以避免误读。

### 3.1 [暂不做] 实物资产管理类

- 装备(individual gear / 公共装备)管理
- 仓库 / 库存 / 出入库流转
- 车辆(救援车 / 后勤车)管理
- 船艇 / 水域装备管理

**为什么**:实物资产管理涉及编号体系、出入库流程、责任人交接、报废流程,是独立业务域,与"基础数据"耦合度低。

### 3.2 [暂不做] 财务相关

- 财务凭证、报销流程
- 捐赠登记、收支管理
- 经费分摊、预算管理

**为什么**:财务有合规要求,不能用通用底座草草承载,必须独立立项。

### 3.3 [暂不做] 复杂救援作业

- 救援任务调度、指挥派单
- 实时定位、轨迹记录
- 现场指挥端、对讲集成

**为什么**:这是"实时作战"业务,对延迟、可用性、移动端体验要求与"内部管理系统"完全不同。

### 3.4 [暂不做] 数据展示类

- 数据大屏
- BI / OLAP 报表
- 复杂统计看板

**为什么**:数据展示依赖"业务数据已经稳定积累",在底座阶段做没意义。

### 3.5 [暂不做] 评价 / 激励 / 奖惩

- 积分系统、排名
- 评优自动计算
- 奖惩流程、纪律记录的自动化

**为什么**:涉及组织文化与人事敏感性,不能由 AI 在底座阶段拍脑袋。

### 3.6 [暂不做] 流程引擎类

- 通用审批流(请假、报销、入队、退队的多级审批)
- 工作流引擎、状态机框架

**为什么**:审批流是"业务模式",不是"基础数据"。即便未来需要,也优先用具体业务表 + 简单状态字段实现,不引入通用流程引擎。

### 3.7 [暂不做] 外部渠道集成

- 微信小程序、公众号深度集成
- 企业微信集成
- 对外公开报名系统(任何"非登录用户能直接提交数据"的入口)

**为什么**:对外渠道有独立的鉴权与防滥用要求,且会牵动"实名认证 / 风控 / 用户协议"等合规面。

### 3.8 [暂不做] 消息通知系统

- 短信通知
- 邮件通知
- 微信 / 企业微信消息推送
- 站内消息中心

**为什么**:通知系统涉及第三方账号、模板审批、配额管理,且与"基础数据"无强依赖。

### 3.9 [暂不做] 自动化校验 / 提醒

- 证书自动校验(对接外部权威库)
- 保险自动校验
- 证件 / 资质到期自动提醒

**为什么**:自动化提醒依赖定时任务 / 队列(v1 §1 已禁),且"自动校验"涉及外部接口选型。

### 3.10 [暂不做] 文件上传具体实现

- 本地文件存储 Provider
- 阿里云 OSS / 腾讯云 COS / Cloudflare R2 等 Provider 接入
- 上传分片、断点续传、CDN

**允许**:在 V2 草案中**讨论附件元数据表**;**不允许**实装任何 Provider。沿用 v1 §1 与 CLAUDE.md §1 的 storage 仅 interface 政策。

### 3.11 [暂不做] 权限模型升级

- RBAC / `permissions` 表 / `user_roles` 多对多
- CASL 等权限框架
- 按钮级权限、字段级权限
- **通用**数据范围权限(如"只能看本部门的人")的**引擎化 / 框架化**

**当前权限模型**:沿用 v1 三层 `Role`(SUPER_ADMIN / ADMIN / USER) + Service 层 `assertCanManageUser`。**禁止** V2 阶段任何形式的"权限框架"扩张。

**允许的事**(避免误读):

- V2 新增模块**允许**在具体 Service 中保留**显式**的范围校验点,沿用 v1 的命名风格,例如 `assertCanManageMember` / `assertCanViewMemberProfile` / `assertCanManageOrganization` 等
- 这些校验点是**单点函数**,实现细节硬编码在对应 Service 内,只服务于本模块的业务边界

**禁止的事**:

- 把多个 `assertCanManageXxx` 抽象为通用 `PermissionService` / `AbilityService` / 装饰器框架
- 引入"权限规则配置表 / 策略 DSL / 表达式求值器"
- 把校验逻辑搬到拦截器 / Guard 做"统一权限层"

> 说明:救援队场景下"部门负责人只能管本部门成员"是真实需求,但 V2 第一阶段**不**通过 RBAC / 数据范围权限引擎实现,**继续走 Service 层显式判断**。是否升级为通用引擎留给未来里程碑评估。

### 3.12 [暂不做] 性能 / 缓存优化

- Redis 接入
- 队列(BullMQ 等)
- 定时任务(Cron 等)
- 用户状态缓存、字典缓存等任何"主动优化"

**为什么**:沿用 v1 §1 与 CLAUDE.md §1 的"不主动加缓存"政策。在没有真实压测数据前,优化是负价值。

### 3.13 [暂不做] AI / 智能化

- LLM 接入(模板的 `modules/ai/` 占位继续保持)
- 向量检索、pgvector
- 智能推荐、智能匹配

### 3.14 [暂不做] 多租户

- 多组织数据隔离
- 租户级配置、计费

**为什么**:srvf-nest-api 服务于单一组织(深圳公益救援队),不存在多租户场景。

### 3.15 [暂不做] 历史数据导入工具

允许在研究文档中**记录"未来会有历史数据导入需求"作为待调研项**(详见 §4.7),但:

- **不**实现导入工具 / 脚本
- **不**为导入做 schema 妥协(例如增加"原始数据 JSON 备份字段")
- **不**评估具体的导入数据格式

### 3.16 出勤 / 考勤 / 训练管理 — 措辞澄清(非简单"不做")

以下两类**业务流程**[暂不做],但**底座研究**允许:

- **考勤业务流程**[暂不做]:不实现"签到打卡 / 请假申请 / 缺勤统计 / 出勤率自动计算"
- **训练业务模块**[暂不做]:不实现"训练计划编排 / 训练成绩录入 / 训练资格认定"

**但允许研究的事**:

- 通用 `events` / `event_participants` 是否能作为**底座模型**承载未来的考勤、训练、活动等场景(详见 §2.6 与 §4.6)
- 训练可作为未来 `eventType` 字典的一种**候选取值场景**(仅作场景假设,不预设字典内容)
- `event_participants` 是否承载"报名 / 签到 / 签退 / 请假 / 缺勤 / 时长统计"等参与状态,**[待调研]**

**红线**:研究允许讨论"底座是否能承载",**实现**仍然只能做最小通用骨架,不能落任何"考勤""训练"专属字段。

## 4. 必须先调研的事(待调研清单)

> 本节列出"在写 `srvf-foundation-data-model-draft.md` 之前必须有阶段性结论"的开放问题。
> 调研可以是用户访谈深圳救援队、查合规规定、对照同类组织实践,也可以是用户直接给出经验判断。
> 每个 [待调研] 项需要的最低产出:**一个能写进草案文档的结论**(可以是"暂时不做""倾向 A 方案""留作运营录入"等)。

### 4.1 [待调研] 真实组织结构

- 当前部门列表(只为校准模型形态,**不**写进任何文档)
- 层级深度(2 层 / 3 层 / 更深?)
- 是否有"小组"或更细粒度
- 是否存在"跨部门小组"或"临时编组"
- 是否存在并列的"职能线"与"业务线"两套树

**结论形态**:"V2 底座支持 N 层树即可""是否需要支持多树根""是否需要支持节点跨树移动"。

### 4.2 [待调研] 队员等级 / 资质体系

- 实际等级取值(预备 / 正式 / 骨干?或别的体系?)
- 等级与资质(如"水域 OW""绳索 R1")的关系:是同一维度还是正交两套?
- 等级是否有互斥关系(同时只能有一个等级)
- 是否需要保留"等级历史"(晋升轨迹)

**结论形态**:"等级走字典""资质走另一套字典或独立表""是否需要历史版本化"。

### 4.3 [待调研] 敏感信息合规面

涉及个人敏感信息字段的范围与合规要求:

- 身份证号:是否存储?如存储,是否加密 / 脱敏 / 仅末四位?
- 紧急联系人(姓名 + 电话 + 关系):是否存储?访问权限如何?
- 医疗信息(过敏史 / 慢性病 / 血型):是否存储?是否分级访问?
- 银行卡 / 支付信息:**[已确认]** 不存储(财务相关 [暂不做],详见 §3.2)
- 证件照片 / 体检报告:**[暂不做]** 实装上传,但讨论附件元数据时是否预留分级访问字段

**结论形态**:"V2 是否实装这些字段""哪些字段进 members 主表 / 哪些进 member_profiles 扩展资料 / 哪些完全延后""读写权限如何设计"。

**[已确认] 数据最小化原则(本节硬约束)**:

任何敏感信息字段进入 V2 草案前,**必须**同时回答以下三问:

1. **业务用途**:这个字段在 V2 / 第一阶段有什么具体业务**已经在用**?(不是"以后可能会用")
2. **查看角色**:谁能读它?谁不能读?读取需不需要二次校验或脱敏?
3. **保存期限**:留多久?队员退队后是否需要清理 / 归档 / 匿名化?

三问**任一未答**或答案是"不知道 / 以后再说" → **该字段不得进入 V2 字段草案**,作为 [暂不做] 项延后。

**禁止的反模式**:"先把字段加进去占位,反正以后会用到""先存着,使用规则以后再补"。这两类思路在敏感信息场景下**禁止**,即便字段在技术上无害。

### 4.4 [待调研] 历史 / 版本化需求

- 队员晋升、转部门、改名、改联系方式的历史是否要溯源?
- 如果要,是"专门的 history 表"还是"靠 audit_logs 反推"?
- 历史需求的真实使用场景是什么(年终汇总?事故追溯?对外报送?)

**结论形态**:"V2 是否做版本化""做的话粒度到字段级还是记录级""可以延后到 V3 吗"。

### 4.5 [待调研] 一人多部门

- 是常态(大部分队员同时属于多个部门)还是例外(极少数兼职)?
- 是否区分"主部门"与"兼职部门"?
- 跨部门时,等级 / 资质 / 角色是否在每个部门独立(同一人在 A 部门是组长、在 B 部门是普通成员)?

**结论形态**:"中间表必要性""中间表是否需要 isPrimary / role / joinedAt 等字段"。

### 4.6 [待调研] 通用 events 模型的承载范围

- 是否真的能用一张 `events` 表 + 类型字典覆盖救援出动 / 训练 / 会议 / 公益活动 / 考核等?
- 还是部分场景(如救援出动)字段差异太大,必须独立建模?
- `event_participants` 是否承载报名 / 签到 / 签退 / 请假 / 缺勤 / 时长?如果承载,是用状态机还是离散字段?

**结论形态**:"V2 是否做 events""做的话承载范围""不做的话第一阶段彻底砍掉这两张表"。

### 4.7 [待调研] 历史数据迁移

- 是否存在历史 Excel / 旧系统数据需要导入?
- 大概数据量级?
- 字段对照关系是否已知?
- 是否需要保留原始数据快照?

**结论形态**:**仅记录需求是否存在**,不评估具体导入方案;真到导入阶段独立立项。

### 4.8 [待调研] 字典是否需要"类型元数据"

判断 `dict_types + dict_items` 双表方案是否成立的关键证据:

- 救援队真实使用的字典中,是否存在需要"类型级配置"的(如:此字典是否允许运营自助新增?是否允许停用?是否有排序需求?是否有父子关系?)
- 如果所有字典都是"扁平 + 仅 code/label"的同质结构,双表是过度设计,应回退到单表

**结论形态**:"双表必要性确认""或回退到单表 + 简化结构"。

### 4.9 [待调研] 附件元数据归属

- 附件归属对象的多样性(队员证件 / 活动照片 / 装备图片 / 培训资料)
- 是用"多态外键"(`ownerType + ownerId`)还是"每张业务表自己挂 attachments"?
- 是否需要支持"一个附件被多个对象引用"?

**结论形态**:"V2 选择哪种归属模式""是否在 V2 阶段就支持多对多"。

## 5. 可以先做的纯基建(无业务争议)

> 本节列出"即便 §4 调研结果未出,也不会被推翻形态"的基建。
> **注意**:列在本节**不等于"现在可以开始 coding"** — 仍需先完成 `srvf-foundation-data-model-draft.md` 评审,且本研究文档仍是"调研与设计阶段"。

### 5.1 [当前倾向] 字典基建(空数据)

- 表结构(无论双表还是单表方案)的草案讨论
- 字典使用约定(哪些场景必须走字典,哪些场景禁止字典化)
- 字典 seed 机制(seed 文件结构,但**不**预填具体字典内容)

**Seed 内容红线**:

- 允许在 seed 中**预留字典加载机制**(读取结构化文件 → 写入 dict_types / dict_items 的代码骨架)
- **设计阶段不得**在仓库 seed 中写入深圳救援队**真实**字典项(真实部门、真实等级、真实活动类型、真实证书类型等)
- 仓库内的 seed 示例数据**只能**使用 neutral-demo 值(如 `demo-type-a` / `demo-item-1`)或**完全留空**
- 真实字典内容由用户在调研评审后**私下提供**,通过环境变量 / 私有 seed 文件 / 运营后台录入,**不进公共仓库历史**(对应风险表 R13)

### 5.2 [当前倾向] 通用附件元数据表(仅元数据)

- 元数据字段范围讨论(`key` / `originalName` / `mimeType` / `sizeBytes` / `uploaderUserId` / 归属字段)
- **不**实装 Provider
- **不**实装上传接口

### 5.3 [当前倾向] 审计日志表(写入策略渐进)

- 表结构草案讨论
- 写入入口(Service 显式调用 vs Interceptor 自动 — 决策见 §4 / 草案文档)
- **不**强制覆盖所有写操作

### 5.4 [已确认] 软删除规范延续 v1

- 沿用 v1 §10 与 CLAUDE.md §10 的"不引入 Prisma 全局软删除中间件"
- 候选封装:在 PrismaService 之上抽 helper 或 per-service `notDeletedWhere()` — **仍显式调用**
- 哪些表软删 / 哪些表硬删(字典 items / 附件 / 审计 / events 各自的策略)由草案逐表标注

### 5.5 [已确认] 命名规范延续 v1

- 时间字段 `createdAt` / `updatedAt` / `deletedAt`
- 主键 `cuid()` 字符串
- 文件标识 `key`(不叫 path / filename / url)
- 软删除标记 `deletedAt: Date | null`(不引入 `isDeleted` 布尔)
- 字典字段命名约定在草案文档中确定

### 5.6 [已确认] 不动 v1 已交付的 auth / users 业务路由

- 沿用 v1 §6 已交付的 14 个接口路径与契约
- V2 仅**追加**新模块,不改 v1 接口
- **不破坏 v1 接口兼容性**是硬约束:已有响应 DTO 字段不删除、不改名、不改类型、不改语义

**users 与 members 关联方案 [当前倾向]**:

若 V2 需要在 users 与 members 之间建立关联,**优先研究**以下两种候选方案,由草案评审拍板:

- **候选 A**:users 表追加可空外键(如 `memberId` 可空),适合"一个登录账号最多绑定一个队员档案"且关联关系稳定的场景
- **候选 B**:独立关联表(如 `user_member_links`),适合"关联关系本身需要属性(绑定时间、绑定人、解绑历史)""未来可能演化为多对多"的场景

**红线**:无论选哪种方案,都**不得**:

- 删除或重命名 v1 `users` 已有字段
- 修改 v1 `users` 接口的请求 / 响应 DTO 字段集
- 在 v1 `users` 接口的响应中新增"必返"字段(可以新增 [可选返回] 字段,但需在草案中显式说明)
- 把 `members` 字段倒灌进 `UserResponseDto`

## 6. 设计原则(适用 V2 全程)

> 本节原则在 V2 全程生效。任何与本节冲突的草案需要显式说明并触发用户决策。

### 6.1 [已确认] 业务身份与登录身份解耦

users(登录身份)与 members(组织身份)**禁止合并**。
不接受任何"users 加几个字段当 member 用"或"members 加 password 当 user 用"的草案。

### 6.2 [已确认] 主表保持克制,扩展资料外移

`members` 主表只承载"高频读取、低敏感、低变更"的字段(具体字段集 [待调研])。
扩展资料(身份证、紧急联系人、医疗信息、证件附件等)走 `member_profiles` 或专表,主表禁止做大宽表。

### 6.3 [已确认] 字典化优先于硬编码,但有红线

可变枚举(等级 / 部门类型 / 活动类型 / 证书类型)**优先字典**。
**红线**:系统级稳定枚举(`Role` / `UserStatus` / 软删除标记)**禁止**字典化。
判断准则:"如果改这个枚举需要改代码逻辑(if/switch),它就不能字典化"。

### 6.4 [已确认] 中间表优于多值字段

成员↔部门、事件↔参与者等多对多关系**必须**走中间表,不接受 JSON 数组、逗号分隔字符串、并列字段(department1 / department2)。

### 6.5 [已确认] 软删除作为默认,但逐表评估

所有"承载业务关键数据"的表预留 `deletedAt`。但"是否对该表启用软删除"由草案逐表给出理由,不无脑全开:

- 字典 items:**倾向不物理删除**,优先使用启停标记(`enabled` / `status`);是否保留 `deletedAt` 字段由草案逐表评估 — **[待草案]**
  > 理由:字典项可能被历史业务数据(已结束的活动、已离队队员档案、已归档的审计记录)引用,物理删除会造成历史语义丢失("当年那个等级是什么"无法回溯)。启停标记可阻止新数据引用、不破坏旧数据语义。
- 审计日志:绝不软删除(也不允许任何删除)— **[已确认]**
- 附件元数据:倾向软删除 — **[待草案]**
- members / organizations / events:软删除 — **[当前倾向]**

### 6.6 [已确认] 关键写操作可映射到审计日志

每个 V2 写操作在草案阶段必须回答:"该操作是否需要审计?"
答案可以是"否",但**必须显式回答**,不留白。
**实现层面**:V2 不强制即时接入,允许只搭表 + 接口、业务点逐步接入。

### 6.7 [已确认] 不预先做权限范围引擎

"部门负责人只能管本部门人"等数据范围权限,V2 通过 Service 层显式判断实现,不引入引擎/中间件。
若实现复杂度爆炸,作为 V3 升级路径,不在 V2 内调和。

### 6.8 [已确认] 文档先于实现

V2 任何 schema / migration / 模块代码动作之前,必须完成:

1. 本研究文档评审通过
2. `srvf-foundation-data-model-draft.md` 评审通过
3. `ARCHITECTURE.md §12` 升级为带 schema 锁定的开发蓝图

未走完三步 → **禁止** `prisma migrate` / 新建模块 / 安装新依赖。

## 7. 风险点

| ID | 风险 | 触发场景 | 缓解措施 |
|---|---|---|---|
| **R1** | 字典滥用 | 把不该字典化的稳定枚举(Role / UserStatus / 软删标记)丢进字典 | §6.3 红线 + 草案逐字典说明"为什么字典化" |
| **R2** | 组织树过度灵活 | 同时支持"多树根 + 跨树移动 + 节点版本化"导致查询复杂度爆炸 | §4.1 调研结果先收敛形态;V2 倾向"单根 + 不支持跨树移动" |
| **R3** | 大宽表诱惑 | 紧急联系人 / 医疗信息 / 装备 / 证件等堆入 members 主表 | §6.2 + 草案中 `member_profiles` 显式收纳扩展字段 |
| **R4** | 审计日志写入路径不统一 | 散落在 Service 各处,接入不一致,部分模块漏写 | §6.6 + 草案选定单一写入入口(显式 vs Interceptor) |
| **R5** | 字典缓存提前优化 | 接入 Redis / 内存缓存"以防字典查询慢" | §3.12 + 沿用 v1 §1"不主动加缓存" |
| **R6** | 软删除范围蔓延 | 无脑给所有表加 `deletedAt`,字典 items 等不该软删的表也加 | §6.5 + 草案逐表说明 |
| **R7** | 组织树数据 seed 越权 | seed 把深圳救援队真实部门名写死,污染模板可复用性 | §4.1 调研后由用户决定 seed 内容,**AI 不擅自填充** |
| **R8** | 通用 events 承载边界失控 | events 想吃下所有"按时间发生的事",反而变成另一种大宽表 | §4.6 调研先收敛承载范围;承载不下就砍,不强行通用 |
| **R9** | "顺手"实现 §3 不做项 | AI 在写 V2 草案时把"装备""通知""审批"顺手补上 | CLAUDE.md §18 调研期硬禁止 + §3 不做清单写死 |
| **R10** | v1 接口被破坏 | 为接 members 而修改 users 已交付接口的响应字段 | §5.6 + 草案禁止破坏 v1 §6 接口契约 |
| **R11** | 历史数据迁移塞进 V2 | 调研发现要迁数据,顺势在 V2 实现导入工具 | §3.15 + §4.7 显式延后 |
| **R12** | 敏感信息无合规结论就实装 | 没等 §4.3 调研结果就先把字段落表 | §4.3 + §6.8 文档先于实现,合规未定 → 字段不进草案 |
| **R13** | 字典 seed 内容外泄 | 在公共仓 / 公共文档中暴露救援队真实运营数据 | seed 内容由用户私下提供,文档与仓库示例不出现真实数据 |

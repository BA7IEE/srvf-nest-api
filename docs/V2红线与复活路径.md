## 0. 文档定位

### 0.1 这份文件是什么

- V2 派生项目当前阶段的**红线与复活路径总览**
- 把多个文档中关于"做什么 / 不做什么 / 何时能做"的散落信息整合到一处,降低新会话 AI / 维护者的拼图成本
- **长期维护文档**:每次 release 落地后状态变化时同步更新(沿 baseline 附录 A 范式)

### 0.2 这份文件不是什么

| 不是 | 权威源 |
|---|---|
| 工程铁律(全局) | `CLAUDE.md` / `AGENTS.md` §1-§18 |
| 蓝图与升级路径 | `ARCHITECTURE.md` §1-§12 |
| V2 通用基线(命名 / 段位 / 软删 / 兼容性 / 验收) | `docs/srvf-foundation-baseline.md` |
| V2 调研结论与暂不做清单 | `docs/srvf-foundation-research.md` §3 / §6 |
| schema / API / 实施计划 | `prisma/schema.prisma` / `docs/v2-api-contract.md` / `docs/v2-plan.md` / `TASKS.md` |
| 历史阶段交接 | `docs/handoff/v0.X.0.md` |
| 批次评审决议 | `docs/批次X_API前评审.md` / `docs/批次X_业务确认稿.md` |

本文件**不重抄**上述文档的内容,只承载"当前红线状态 + 复活路径状态"的快照。

### 0.3 冲突优先级

冲突时,按以下顺序让步(沿 `docs/srvf-foundation-baseline.md §14.4`):

1. `ARCHITECTURE.md`(蓝图 + §9 升级路径 + §11 V1.1 加固 + §12 V2)
2. `CLAUDE.md` / `AGENTS.md` §1-§18(工程铁律)
3. `docs/srvf-foundation-baseline.md`(V2 通用基线 A 档 13 项)
4. **本文件**(V2 红线与复活路径总览)
5. `docs/批次X_API前评审.md`(单批次评审稿)

发现本文件与更高优先级文档冲突 → 本文件让步,Claude Code **必须暂停说明**,由用户拍板修订。**禁止** AI 自行调和。

### 0.4 适用范围

本文件聚焦 V2 派生项目(`srvf-nest-api`)。v1 / V1.1 模板侧的红线(`CLAUDE.md §1` / `§17`)沿用,本文件不重述,但作为本文件的隐含基线。

## 1. 五档定义

| 档 | 含义 | 修订门槛 | 举例 |
|---|---|---|---|
| **A 类**<br>真正不可破的红线 | 架构底线;改动需走 `ARCHITECTURE.md §9` 升级路径 + 用户拍板;AI 绝不自行调和 | 极高:用户明确拍板 + 评审 + 升级路径 | audit_logs 写入后不可改不可删 / v1 14 接口零漂移 / Role enum 不可绕评审扩展 |
| **B 类**<br>当前批次禁止项 | **不是永久不做**,只表示本 PR / 本批次范围外;下一批次可重新评估 | 低-中:下一批次 D6 评审或用户拍板单批范围调整 | 批次 6 第一波只迁 8 处 / 批次 5-A 不做 dryRun |
| **C 类**<br>可复活项 | 已识别需求方向 + 触发条件;复活必须走单独批次评审 + 用户拍板;**禁止顺手做** | 中-高:看具体项是业务确认 / schema 评审 / 架构评审 | audit_logs 第二波 / attachments / member_profiles / APD 权限细分 |
| **D 类**<br>历史过期表述 | 旧文档中某些"延后/不做"已被后续 release 局部突破;旧 handoff 不回改,主线文档加范围说明 | 文档级,不影响代码 | `docs/v2-plan.md §7.1` "audit_logs 不开发" 的实际范围限于 V2 第一阶段 Step 1-7 |
| **E 类**<br>待业务确认项 | 不是技术红线,而是业务方未拍板;AI 不替业务决定 | 低:由业务确认稿 / 用户访谈触发 | 是否记录查看行为 / APD 权限边界 / 统计口径 |

**铁律**:

- 把 **B 类**误判成 **A 类** → 不当锁死可演进的范围
- 把 **C 类**误判成 **A 类** → 拒绝合理的业务诉求
- 把 **E 类**误判成 **A 类** → 把业务问题包装成技术禁令
- 把 **A 类**误判成 **B 类 / C 类** → 红线破口

## 8. 修订纪律

### 8.1 每次 release 后同步更新

某批次落地导致状态变化时,本文件应同步:

- **新批次禁止项**:加入 §3 B 类(本批次 F1-FN 摘要,完整版仍以批次评审稿为准)
- **已完成的 C 类**:从 §4 移除或更新状态行
- **新过期的措辞**:加入 §5 D 类对照表
- **新业务确认项**:加入 §6 E 类

### 8.2 不回改历史 handoff

- `docs/handoff/v0.X.0.md` 是阶段快照,**不回头修改**
- 状态枢轴(audit_logs 从延后到局部启动等)在本文件 §5 D 类描述

### 8.3 把"本批次不做"改"可复活"必须有评审依据

- 不能仅凭 AI 或维护者直觉降级
- 需要至少一项:**D6 评审稿的明确决议** / **用户拍板的明确意见** / **批次评审稿状态调整说明**(沿 5-A D6 v1.1 §2.5 范式)

### 8.4 修订 commit 前缀

- `docs(v2-redlines): <章节> <简述>`(本文件单独修订)
- 主线文档(README / ARCHITECTURE / v2-plan / TASKS)同步小修订时,沿对应文档的修订前缀

### 8.5 与其他文档的协同更新

本文件修订时,如果触发更高优先级文档需要同步(`ARCHITECTURE.md §12.8.4` 范围说明 / `docs/v2-plan.md §7.1` 范围说明 / `TASKS.md §5.5.4.3` 状态行等):

- 在**同一 PR 内**完成同步小修订(避免拆 PR 后状态分裂)
- 本文件**不替代**上述文档,只在变更时同步小修订

### 8.6 与 baseline §1.1 段位映射的关系

本文件**不重抄**段位映射;BizCode 段位映射的单一权威源是 `docs/srvf-foundation-baseline.md §1.1`。本文件涉及段位时,仅引用,不重抄。

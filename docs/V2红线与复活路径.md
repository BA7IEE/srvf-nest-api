## 8. 修订纪律

### 8.1 每次 release 后同步更新

某批次落地导致状态变化时,本文件应同步:

- **新批次禁止项**:加入 §3 B 类(本批次 F1-FN 摘要,完整版仍以批次评审稿为准)
- **已完成的 C 类**:从 §4 移除或更新状态行
- **新过期的措辞**:加入 §5 D 类对照表
- **新业务确认项**:加入 §6 E 类

### 8.2 不回改历史 handoff

- `docs/handoff/v0.X.0.md` 是阶段快照,**不回头修改**
- 状态枢轴(audit_logs 从延后到局部启动等)在本文件 §5 D 类描述

### 8.3 把"本批次不做"改"可复活"必须有评审依据

- 不能仅凭 AI 或维护者直觉降级
- 需要至少一项:**D6 评审稿的明确决议** / **用户拍板的明确意见** / **批次评审稿状态调整说明**(沿 5-A D6 v1.1 §2.5 范式)

### 8.4 修订 commit 前缀

- `docs(v2-redlines): <章节> <简述>`(本文件单独修订)
- 主线文档(README / ARCHITECTURE / v2-plan / TASKS)同步小修订时,沿对应文档的修订前缀

### 8.5 与其他文档的协同更新

本文件修订时,如果触发更高优先级文档需要同步(`docs/archive/plans/architecture-v2-first-stage-blueprint.md §12.8.4` 范围说明等;原 `ARCHITECTURE.md §12.8.4`,PR-6 已归档;原 `docs/v2-plan.md §7.1` 范围说明已归档于 `docs/archive/plans/v2-first-stage-plan.md`,PR-5;原 `TASKS.md §5.5.4.3` 已归档于 `docs/archive/plans/v2-design-phase/tasks.md`,本文件 §4.3 已成为 V2.x 复活的 active 权威源):

- 在**同一 PR 内**完成同步小修订(避免拆 PR 后状态分裂)
- 本文件**不替代**上述文档,只在变更时同步小修订

### 8.6 与 baseline §1.1 段位映射的关系

本文件**不重抄**段位映射;BizCode 段位映射的单一权威源是 `docs/srvf-foundation-baseline.md §1.1`。本文件涉及段位时,仅引用,不重抄。

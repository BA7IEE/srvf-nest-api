### 4.4 StorageProviderRouter 动态路由(沿 Q-89-1)

- `STORAGE_PROVIDER` DI token(Symbol)= `useExisting StorageProviderRouter`
- 每次 Provider 方法调用都通过 Router `resolve()` 决定 LocalProvider / CosProvider
- `resolve()` 内部读 `StorageSettingsService.getResolved()`(含 60s 缓存 + 解密)
- **运维改 `storage_settings.providerType`(LOCAL ↔ COS)≤ 60 秒内自动切换**;**无需重启服务**
- 切换路径:`PATCH /storage-settings { providerType: 'COS' }` → `StorageSettingsService.invalidate()` 主动失效缓存 → 下一次 resolve 拿新 providerType
- Symbol DI token 而非字符串(规避 typo / 多 Provider 竞争注册)

### 4.6 AES-256-GCM 凭证加密(沿 §6.6.1 + Q21)

- **算法**:AES-256-GCM
- **派生 key**:Node 原生 `scrypt(STORAGE_ENCRYPTION_KEY, salt='storage-settings', keyLen=32)`
- **IV**:每次加密用 `randomBytes(12)`(12 字节)
- **authTag**:16 字节(由 `cipher.getAuthTag()` 取)
- **payload 格式**:`[IV(12)][authTag(16)][ciphertext]`,集中存 `storage_settings.secretIdEncrypted` / `secretKeyEncrypted` 两列
- **`STORAGE_ENCRYPTION_KEY`** env:
  - production 启动 fail-fast(沿 v1 §14 / CLAUDE.md §0 启动强校验)
  - dev / test 允许缺失 → `StorageCryptoService.isAvailable = false`,`StorageSettingsService.getResolved()` 返 `credentialStatus = MISSING`
  - 加密 key 与凭证密文**分离存储**(沿 Q23 例外:加密 key 长期存 env,凭证密文长期存 DB;不混)

### 4.8 `POST /upload-url` + `POST /confirm-upload`(沿 §8.3 + §8.4 + Q6 / Q7)

**模式 B 签名直传**(沿 F2):

- `POST /api/v2/attachments/upload-url`:入参 5 字段(`ownerType` / `ownerId` / `originalName` / `mime` / `sizeBytes`);出参 6 字段(`key` / `uploadUrl` / `uploadHeaders` / `uploadMethod` / `expiresAt` / `uploadToken`)
- `POST /api/v2/attachments/confirm-upload`:入参 1 必填(`uploadToken`)+ 1 可选(`checksum` sha256 64-hex);出参 `AttachmentResponseDto`(沿现有 PR #6b 结构)
- **`uploadToken` HMAC-SHA256 紧凑格式**:`<base64url(claims)>.<base64url(hmac)>`(类 JWT 风格;**0 jsonwebtoken 依赖**;沿 §8.3.4 + Q-91-X 拍板)
- **复用 `STORAGE_ENCRYPTION_KEY`** 作 HMAC secret(节省 env 个数;不与加密 key 混淆用途:同一 env 既派生 AES-256-GCM key 又作 HMAC secret,加密路径与签名路径互不污染)
- **confirmUpload 6 步流程**(沿 Q7):
  1. **验签**(timingSafeEqual)+ 解 claims(`key` / `ownerType` / `ownerId` / `mime` / `sizeBytes` / `exp` / `uploaderId`)
  2. **headObject** Provider 真实文件(取 `size` / `etag` / `lastModified`)
  3. **size 一致性**(headObject `size` 必须等于 claims `sizeBytes`;不一致抛 `ATTACHMENT_SIZE_EXCEEDED`)
  4. **PII 不重做**(沿 Q10:upload-url 已检,confirm-upload 不接受 originalName 等 PII 字段重传)
  5. **落库 + audit fail-fast**(同事务 `$transaction(async (tx) => { tx.attachment.create(...); auditLogs.log({ tx, event: 'attachment.upload', extra: { ..., uploadConfirmedAt, uploadVia: 'direct' } }) })`;沿 B4 extra 扩展)
  6. **`generateDownloadUrl` 填 accessUrl** 后返回

**反 replay 防御**:依赖 **`Attachment.key @unique`**(P1 技术债 PR #92)+ **P2002 catch**(并发 race 兜底)+ **Service `findFirst` 早返**(串行场景省事务开销);三层防御覆盖。

### 4.9 后台 Storage Settings 管理(沿 §6.5 + §6.6 + Q-11)

3 端点(全部 `@Roles(SUPER_ADMIN, ADMIN)` 入口;**不接 `rbac.can()`**;沿 C-7 配置三表范式 F4):

| 端点 | 用途 | 关键设计 |
|---|---|---|
| `GET /api/v2/storage-settings` | 读当前配置(凭证字段返 `credentialStatus`)| **凭证 6 层防护**:response 不返密文 / 不返明文;`credentialStatus` 三态化 |
| `PATCH /api/v2/storage-settings` | 改非凭证字段(providerType / bucket / region / endpoint / keyPrefix / TTL / mimePolicyMode 等)| **PATCH upsert**(行不存在自动创建 default;沿 Q-11-1);**禁止 PATCH 凭证字段**(沿 Q-11-X;forbidNonWhitelisted) |
| `POST /api/v2/storage-settings/reset-credentials` | 重置凭证(`secretId` + `secretKey` 入参 → AES-256-GCM 加密落库)| **响应不回显**(沿 Q22);response 仅返 `credentialStatus: CONFIGURED`;Service 主动 `invalidate()` 缓存 |

**`credentialStatus` 三态语义**:

- `CONFIGURED`:`secretIdEncrypted` + `secretKeyEncrypted` 均就位,且解密成功
- `MISSING`:`secretIdEncrypted` 或 `secretKeyEncrypted` 缺一(初始态)
- `INVALID`:密文存在但解密失败(`STORAGE_ENCRYPTION_KEY` 已轮换 / 密文损坏)

**凭证 6 层安全防护**(沿 §6.6;`secretId` / `secretKey` 明文从入口到落库全程隔离):

1. **response 隔离**:`StorageSettingsResponseDto` 不含 `secretIdEncrypted` / `secretKeyEncrypted` / `secretId` / `secretKey` 任何形式
2. **日志屏蔽**:`AppConfig.LOG_REDACT_KEYS` 沿用 v1.1 范式(`secretId` / `secretKey` / `password` / `token` 等命中即 `[REDACTED]`)
3. **DB 密文**:DB 列名 `secretIdEncrypted` / `secretKeyEncrypted` 显式 `Bytes` 类型;**永不**存明文列
4. **IV 随机**:每次加密 `randomBytes(12)`;同明文 → 不同密文(避免 chosen-plaintext 攻击)
5. **forbidNonWhitelisted**:全局 ValidationPipe(沿 v1 §7)拒绝任何未在 `UpdateStorageSettingsDto` 白名单的字段(如客户端传 `secretId` 字段会直接抛 BadRequest)
6. **DTO 字段集**:`UpdateStorageSettingsDto` / `ResetStorageCredentialsDto` 完全分离;`Reset` 走独立端点,无法通过 PATCH 偷渡

**30 e2e 覆盖**(`storage-settings.e2e-spec.ts`):
- GET:三态 credentialStatus / 凭证字段不外露 / RBAC 入口 / 缓存命中 + 缓存失效
- PATCH:upsert 路径 + 字段白名单 + 凭证字段拒绝 + 缓存失效 + RBAC 入口
- reset-credentials:成功路径 + 凭证解密往返 + IV 随机性 + RBAC 入口 + response 不回显 + 缓存失效

## 9. 已知 quirk:`gh pr merge` worktree checkout main 冲突预案

本仓库在 `.claude/worktrees/` 下创建临时 worktree 时,`gh pr merge --squash --delete-branch` 会撞:

```
failed to run git: fatal: 'main' is already used by worktree at '/Users/.../srvf-nest-api'
```

**根因**:主仓 worktree 占用 `main` 分支,gh 试图本地 checkout main 失败。

**预案**(沿 PR #54-#95 累计 ≥ 20 次验证成功):

```bash
# 1. 仍执行 gh pr merge(GitHub 端会完成 squash merge)
gh pr merge <PR#> --squash --delete-branch
# → 报 worktree 错误,但 GitHub 端已 MERGED

# 2. 验证 GitHub 端 MERGED 状态
gh pr view <PR#> --json state,mergedAt,mergeCommit

# 3. 检查远端分支是否已删(--delete-branch 因 worktree 错误未执行)
git ls-remote --heads origin <branch-name>

# 4. 手动删除远端分支
git push origin --delete <branch-name>

# 5. 主仓库 ff main 到最新
git -C /Users/dengwang/Documents/coding/srvf-nest-api pull --ff-only origin main

# 6. worktree 同步到 main(开下一 PR 前)
git fetch origin main && git reset --hard origin/main
```

详细步骤见 v0.8.1 handoff §11(预案不变;PR #54-#95 全部按此流程操作)。

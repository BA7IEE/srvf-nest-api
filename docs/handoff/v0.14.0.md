## 4. P0-E 实装事实(v0.14.0 落地后)

### 4.1 新增 3 个 API 端点(沿 PR #127 / 评审稿 §4)

| 端点 | 鉴权 | 限流 | 入参 DTO | 主响应 | 行为 |
|---|---|---|---|---|---|
| `POST /api/auth/refresh` | `@Public()` | `@RefreshThrottle()`(IP 30/60)| `RefreshTokenDto { refreshToken }`(严格白名单 1 字段)| 与 login 同样的 5 字段(`accessToken / userId / username / refreshToken / refreshExpiresAt`)| rotation always:发新 refresh + 旧 refresh 同事务标 `rotatedAt + revokedAt + replacedById`;reuse detection 触发 family revoke;absolute expiration 不延长 |
| `POST /api/auth/logout` | `@Public()` | 无限流(刻意;沿评审稿 §3.7) | `LogoutDto { refreshToken }`(严格白名单 1 字段) | 200 + `data: null` | 幂等:不存在 / 已撤销 / 已过期 → 仍返 200;只撤销当前 row,不动 family 其他 row;access token 不消费、不吊销 |
| `POST /api/auth/logout-all` | `JwtAuthGuard` | `@PasswordChangeThrottle()`(复用,IP 5/60)| 无 | `{ revokedCount: number }` | `updateMany` 撤销该 user 全部未过期未撤销 refresh,`revokedReason='logout'` |

### 4.2 扩展 `POST /api/auth/login`

- `LoginDto` 入参 schema **严格 zero drift**(沿评审稿 D-1):字段名 / 类型 / `@Matches` / `@MinLength` / `@MaxLength` 全保留;**禁止**新增任何字段(`rememberMe` / `deviceId` / `clientId` / `keepSignedIn` 全不接)
- `LoginResponseDto` 字段集恰好 **5 项**:`accessToken` / `userId` / `username` + **新增** `refreshToken` / `refreshExpiresAt`
- `refreshExpiresAt` 是 **ISO 8601 UTC 字符串**(`new Date(now + ttlMs).toISOString()`,示例 `"2026-08-16T00:00:00.000Z"`),语义是 **refresh token family 的 absolute expiration 时刻**;rotation 后新 refresh token **继承同一个时刻字符串**;**禁止** sliding expiration

### 4.3 JWT 与 refresh token 行为锁定

- **JWT payload 严格 zero drift**:仍 `{ sub, username }`(+ 标准 `iat / exp / nbf`);**禁止**新增 `role` / `permissions` / `tokenVersion` / `tv` / `jti` / `email` / 任何业务字段
- `JwtStrategy.validate` 严格保持 `select: { id, username, role, status, memberId }`,仅校验 `deletedAt === null && status === ACTIVE`;**禁止**读 `passwordHash` / `tokenVersion`(后者本期不存在)
- **refresh token 生成**:`crypto.randomBytes(32).toString('base64url')`(256 bit 熵);**禁止** JWT / UUID / `Math.random`
- **refresh token 存储**:**明文绝不入库**;DB 仅存 `tokenHash = crypto.createHash('sha256').update(raw).digest('hex')`(64 字符 hex);字段 `tokenHash @unique`
- **禁止**用 bcrypt / argon2 哈希 refresh token(高熵随机串无暴破语义,sha256 sub-ms 性能远优)
- **refresh token 明文绝不进入**:日志 / audit `context.*` / OpenAPI 示例 / 测试 fixture / 测试快照 / 文档示例 / handoff / release notes;只在 login / refresh 接口响应体 `data.refreshToken` 中出现一次

### 4.4 新增 schema(沿 PR #127)

**migration**:`prisma/migrations/20260517165220_add_refresh_tokens` — `refresh_tokens` 表:

```
id              cuid PK
userId          FK users.id
tokenHash       String @unique (sha256 hex 64 chars)
familyId        String (cuid, 同一 login chain 共享)
expiresAt       DateTime (family absolute expiration; rotation 不延长)
createdAt       DateTime @default(now())
rotatedAt       DateTime? (rotation 时设置;raw 已被新 refresh 取代)
revokedAt       DateTime?
revokedReason   String? (logout | self-password-change | admin-password-reset | admin-disable | admin-delete | family-revoked)
replacedById    String? @unique (chain 向后指向新 row id)
ipFirstSeen     String?
uaFirstSeen     String?

6 索引 + 2 FK(userId / replacedById);0 修改既有表;User 仅追加反向 relation 不增字段
```

### 4.5 联动撤销 4 场景(沿评审稿 §7 + `CLAUDE.md §9`)

| 场景 | 触发接口 | 撤销范围 | `revokedReason` | audit |
|---|---|---|---|---|
| 本人改密 | `PUT /api/users/me/password` | 该 user 全部未过期未撤销 refresh | `self-password-change` | `password.change.self` extra 加 `refreshTokensRevoked: count` |
| 管理员重置 | `PUT /api/users/:id/password` | 目标 user 全部 refresh | `admin-password-reset` | `password.reset.by-admin`(P0-E 顺手补) extra.refreshTokensRevoked |
| 用户被禁用 | `PATCH /api/users/:id/status` → `DISABLED` | 目标 user 全部 refresh | `admin-disable` | 仅撤销 refresh,**不补新 audit**(沿 D-PR3-2) |
| 用户被软删 | `DELETE /api/users/:id` | 目标 user 全部 refresh | `admin-delete` | 仅撤销 refresh,**不补新 audit** |

**四场景的 `updateMany` 必须在同一 `prisma.$transaction` 内与主写操作执行**(沿 P0-D `prisma.$transaction` 范式)。

### 4.6 access token 行为锁定(沿评审稿 D-4)

- **本期不主动吊销** access token;依赖 `JWT_EXPIRES_IN=15m` 自然过期 + `JwtStrategy.validate` 每请求查库阻断 `DISABLED` / 软删用户
- access token blacklist / JWT revoke list **本期不做**(沿 §1 C 档)
- `tokenVersion` 字段 **本期不做**(沿 D-4 + §1 B 档;真出现"改密后所有 access 立即失效"诉求时单独评审)
- `users-change-my-password.e2e-spec.ts §7.5` 反向锁定断言**继续保留**(改密后旧 access token 仍可调 `/me`)

## 6. 运维升级注意(生产环境上线必读)

### 6.1 env 变量(必须在 `prisma migrate deploy` 之前注入)

| env | 必填 | 推荐值 | 说明 |
|---|---|---|---|
| `JWT_EXPIRES_IN` | **必须** | `15m` | access token TTL;v0.13.0 默认 `7d`,v0.14.0 起强收敛到 `15m`(沿评审稿 D-5) |
| `JWT_REFRESH_EXPIRES_IN` | **必须** | `90d` | refresh token absolute expiration;**未设置会启动 fail-fast**(`jwt.config.ts` zod schema 校验) |
| `REFRESH_THROTTLE_LIMIT` | 可选 | `30` | refresh 接口限流次数;默认 30,推荐区间 [1, 100] |
| `REFRESH_THROTTLE_TTL_SECONDS` | 可选 | `60` | refresh 限流窗口秒数;默认 60,推荐区间 [1, 3600] |

**fail-fast 行为**:`JWT_REFRESH_EXPIRES_IN` 未设置 / 格式错误(非 `ms` 格式如 `"90"` `"abc"`)/ `REFRESH_THROTTLE_*` 越界 → 应用启动失败,Pod 不进入 ready,K8s 重启循环。**上线前必须在 staging / production secret manager 中预置这 4 个 env**(`REFRESH_THROTTLE_*` 不设也可,走默认值)。

### 6.2 数据库迁移

```bash
# production 环境(沿 baseline §9 + v0.12.0 §7.2 不变)
DATABASE_URL=... pnpm prisma migrate deploy
```

**会应用的 migration**:`20260517165220_add_refresh_tokens`

- 创建 `refresh_tokens` 表 + 6 索引 + 2 FK
- **0 修改既有表 / 0 数据回填 / 0 DROP**
- 用户表 / 其他业务表数据完全不动
- 上线 5 分钟内 sub-ms 完成(单 CREATE TABLE + 索引 / 千万级 user 表无 ALTER 风险)

### 6.3 上线顺序锁定

```
1. 应用容器停止旧版本
2. 注入 env(必须先于 migrate deploy):
     JWT_EXPIRES_IN=15m
     JWT_REFRESH_EXPIRES_IN=90d
     [REFRESH_THROTTLE_LIMIT=30(可选)]
     [REFRESH_THROTTLE_TTL_SECONDS=60(可选)]
3. pnpm prisma migrate deploy  ← 必须 env 已注入,否则下一步启动 fail-fast
4. 应用容器启动新版本
5. health/live + health/ready 双绿
6. 切流量
```

**顺序倒了的后果**:env 没注入就启动 → 容器 fail-fast → Pod 反复重启;migrate 未跑就启动 → `refresh_tokens` 表不存在 → login 试图 `INSERT INTO refresh_tokens` 失败 → 5xx 雪崩。

### 6.4 access TTL 变更影响(`7d` → `15m`)

- **客户端必须实施 refresh 流程**:access token 15m 自然过期后,客户端拿响应里的 `refreshToken` 调 `POST /api/auth/refresh` 换新一对;沿评审稿 §4 + [`docs/first-release-frontend-scope.md §3.2.1`](../first-release-frontend-scope.md) token 生命周期段
- **旧 access token 自然过期**:升级当日已签发的 v0.13.0 access token(7d TTL)在客户端侧依然按签发时刻 + 7d 失效(JWT 自包含 `exp`);但**新签发**的 access token 立即 15m;若客户端不实施 refresh,15m 后必须重新登录
- **客户端实施前的过渡期**:可临时把 `JWT_EXPIRES_IN` 调回更长(env 可热改,改后重启容器即生效);但不建议长期保留长 access TTL,会让 DISABLED / 软删用户即时阻断窗口拉长

### 10.2 v0.14.0 段内新增 / 强化的红线(P0-E)

沿 `CLAUDE.md §9 P0-E refresh token 鉴权铁律` 16 类硬约束 + [P0-E 评审稿 v1 §3 / §5 / §7](../first-release-p0e-refresh-token-review.md):

- **refresh token 生成锁死**:`crypto.randomBytes(32).toString('base64url')`;**禁止** JWT / UUID / `Math.random`
- **refresh token 存储锁死**:DB 仅存 `sha256(raw)` hex;**禁止** bcrypt / argon2 / 明文 / 截断 hash
- **明文绝不进入**:日志 / audit / OpenAPI 示例 / 测试 fixture / 快照 / 文档 / handoff / release notes;只在 login / refresh 响应体 `data.refreshToken` 中出现一次
- **`LoginDto` 入参 zero drift**:**禁止**新增任何字段(包括 `rememberMe` / `deviceId` / `clientId`)
- **JWT payload zero drift**:严格 `{ sub, username }`;**禁止**塞 `role` / `permissions` / `tokenVersion` / `tv` / `jti` / `email`
- **`LoginResponseDto` 字段集锁死**:恰好 5 项(`accessToken / userId / username / refreshToken / refreshExpiresAt`);**禁止**再扩
- **rotation always**:每次 refresh 必发新 + 旧 row 同事务标 `rotatedAt + revokedAt + replacedById`
- **absolute expiration**:rotation 不延长 `expiresAt`,严格继承 family 首个 token 的时刻;**禁止** sliding / refresh-on-use 延期
- **reuse detection 触发 family revoke**:`refresh` 收到 `rotatedAt != null` 的 row → 同事务 `updateMany({ familyId, revokedAt: null }, revokedReason: 'family-revoked')` 然后抛 `REFRESH_TOKEN_INVALID`
- **`refreshExpiresAt` 是 ISO 8601 UTC 字符串,不是 TTL**:`new Date(now + ttlMs).toISOString()`;**禁止**回传 `"90d"` / 数字毫秒
- **logout 幂等**:不存在 / 已撤销 / 已过期 → 仍返 200;**禁止**抛业务码
- **logout 不消费 / 不吊销 access token**(即便 access 随头传入,也不校验)
- **logout-all 走 `JwtAuthGuard`**;**禁止**改成 `@Public()`
- **联动撤销 4 场景同事务执行**:改密 / 重置 / 禁用 / 软删的 `updateMany` 必须在主写操作的 `prisma.$transaction` 内,**禁止**异步 / 队列 / fire-and-forget
- **三 throttler 物理隔离**:`default` / `password-change` / `refresh` 在 `throttle-options.ts` 分别注册;`ThrottlerBizGuard` 按 `throttler.name` 与 metadata 对应放行非匹配 throttler;**禁止**复用 `@LoginThrottle` 或 `@PasswordChangeThrottle` 装饰 refresh 接口
- **限流参数不硬编码**:从 `app.config.ts.refreshThrottle.{limit, ttlSeconds}` 注入;**禁止**写死 30 / 60 在装饰器
- **失败码不拆细**:`REFRESH_TOKEN_INVALID=10007` 唯一;**禁止**新增 `EXPIRED` / `REVOKED` / `REPLAY`(沿 v1 §8 防账号枚举)
- **access token 仍不主动吊销**(沿 D-4);**禁止**"顺手"加 access blacklist / `tokenVersion`;`tokenVersion` 真要做时单独评审,沿 §1 B 档路径
- **audit `extra` 禁止写**:refresh token 明文 / `tokenHash` / `passwordHash` / IP 完整段(IP 已在 `AuditContext.ip` 字段)
- **`JWT_EXPIRES_IN=15m` 与 `JWT_REFRESH_EXPIRES_IN=90d` 是部署必需 env**:未设置 / 格式错 → `jwt.config.ts` fail-fast;**禁止**用默认值兜底(沿 v1 §14 启动强校验铁律)
- **migration 100% additive**:`20260517165220_add_refresh_tokens` 仅 CREATE TABLE;**禁止**回头加 ALTER 既有表 / 数据回填;真要改 `refresh_tokens` schema 时,新建独立 migration
- **`User` 表只追加反向 relation,不增字段**:**禁止**加 `tokenVersion` / `lastRefreshAt` / `refreshTokenCount` 任何字段

### 10.3 v0.14.0 段常见反模式提醒

| 反模式 | 为什么禁止 |
|---|---|
| "顺手"在 `LoginDto` 加 `rememberMe` 决定 refresh TTL 是 30d / 90d / 365d | 违反 D-1 zero drift;TTL 是服务端 env 配置,客户端无权决定;真要做"长记忆"诉求时单独评审 |
| "顺手"在 JWT payload 加 `role` 减少每请求查库 | 违反 D-4 zero drift;每请求查库是有意设计(被禁用户即时失效),沿 v1 §8 升级条件未达成不优化 |
| "顺手"用 `Math.random()` / `Date.now()` 生成 refresh token | 违反 §5.1 + `CLAUDE.md §9 P0-E`;熵不足,可被预测 |
| "顺手"用 bcrypt 哈希 refresh token | 违反 §5.1;高熵随机串无暴破语义,bcrypt 浪费 CPU |
| "顺手"在 audit `extra` 写 `tokenHash` / 部分 refresh raw 前缀 | 违反 §5.9 + `CLAUDE.md §9 P0-E`;hash 也是凭证形态,泄漏会让 DB 备份持有者反推 token |
| "顺手"在 logout 接口校验 access token / 抛 401 | 违反 §4.3;logout 是 `@Public()` 幂等设计,access 过期后仍要能 logout |
| "顺手"加 `Retry-After` / `X-RateLimit-Remaining` 头 | 违反 V1.1 §17.7 + 评审稿 §3.7;泄漏阈值便利攻击者 |
| "顺手"加 sliding expiration("用 refresh 就续期")| 违反 D-5;90d absolute 是 OWASP 红线,无限续期等于无效防御 |
| "顺手"在 reuse detection 时不 revoke family,只返错 | 违反三不变式;reuse 是凭证被盗的强信号,必须断 family |
| "顺手"在改密 / 重置 / 禁用 / 软删后**异步**撤销 refresh | 违反 §7;**必须同事务**,异步 / 队列会留下窗口期被盗 token 仍可 refresh |
| "顺手"启动 P0-F RBAC / P0-H 演练 / P0-I SOP 任何并行 PR | 均需用户授权 + D 档评审;P0-F 涉及 18 个非 attachments 模块 |
| "顺手"打 `v0.14.0` tag / 发 Release | 维护者权限边界;AI 仅 bump + handoff |

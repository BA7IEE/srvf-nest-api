## 6. 批次 5-A 与 ContributionRule 配置链路

```
管理员维护规则(本批次新增):
  POST /api/v2/contribution-rules
    ↓ service 同事务内
    ┌─ dict 校验:activity_type + attendance_role(active)
    ├─ 字段语义校验:pointsAbove != null → durationThreshold != null && > pointsBelow
    ├─ ACTIVE 唯一性兜底:含 durationThreshold = NULL 维度
    └─ P2002 兜底 → 23002
  GET / PATCH / DELETE 同套权限 + 校验组合

attendance 端(本批次不动,沿 batch 4-B):
  POST /api/v2/activities/:id/attendance-sheets
    ↓ AttendancesService.applyContributionRulePrefill(同事务)
    ├─ tx.contributionRule.findMany({ activityTypeCode, attendanceRoleCode, status: ACTIVE, deletedAt: null })
    ├─ 取首条匹配规则(NULL durationThreshold 经 5-A 兜底后退化为"唯一一条")
    └─ 计算 candidatePoints ∩ dailyCap(默认 1.5)
```

核心不变式(代码层强约束,**改 V2 代码前必须保留**):

- 5-A 配置链路:
  - `durationThreshold = NULL` 多条 ACTIVE 由 service `assertActiveUnique` 拒(23002),DB partial unique 在 PG NULL 行为下**不约束** NULL 多条
  - PATCH 不接受 `activityTypeCode` / `attendanceRoleCode` / `durationThreshold`(DTO 白名单 + `forbidNonWhitelisted` 双重防护);改维度走"停用旧 + 新建"
  - 规则修改**只影响新提交** AttendanceSheet,**不重算**任何已存在 Sheet(沿"submit 时同事务内预填,之后不再读")
  - `softDelete` 写 `deletedAt + deletedByUserId`(schema 已在 batch 4-A 含字段);`status` 不强制改 `INACTIVE`
- 批次 4-B 链路(继续生效,**本批次不动**):
  - `AttendanceSheet` 5 态闭集 + APD 终审字段;`attendance.recorded` 仅在 `final-approve` 触发;`approved` = 终审通过(贡献值正式生效)
  - `applyContributionRulePrefill` 三态语义:omit → 预填 / 显式 `null` → 跳过(P2-1 已 e2e 覆盖) / `number` → 不覆盖

### 12.3 ACTIVE 唯一性兜底语义(沿 D6 v1.1 §4.5)

| 场景 | DB partial unique | service `count` |
|---|---|---|
| `durationThreshold = 1.0` 多条 ACTIVE | ✅ P2002 兜底 | ✅ 23002 优先路径 |
| **`durationThreshold = NULL` 多条 ACTIVE** | ❌ PG NULL 行为不约束 | ✅ **23002 唯一兜底来源** |
| 软删后同维度再插 ACTIVE | ✅ 允许(`deletedAt IS NULL` 过滤) | ✅ `count = 0`,允许 |
| `INACTIVE` 重复 | ✅ 允许(`status='ACTIVE'` 过滤) | ✅ `count = 0`,允许 |

**所有 P2002 兜底场景都先经 service `count`,顺序保证 23002 优先,P2002 仅在并发 race 兜底**。

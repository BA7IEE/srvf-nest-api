## 4. 当前能力(v0.13.0 落地后)

### 4.1 P0-D 本人自助改密(沿 PR #117)

新接口:`PUT /api/users/me/password`(任意登录用户;不标 `@Roles`,沿 GET/PATCH `/me` 范式)。

**入参 DTO**(`ChangeMyPasswordDto`,严格白名单 2 字段;`forbidNonWhitelisted` 兜底拒绝额外字段):

| 字段 | 校验 | 校验失败错误码 |
|---|---|---|
| `oldPassword` | `@IsString` + `@IsNotEmpty`;**不**做长度 / 字母 / 数字校验(原因:登录侧 `LoginDto.password` 同样不做,防泄漏密码强度规则) | `BAD_REQUEST=40000` |
| `newPassword` | 与 `ResetUserPasswordDto.newPassword` **完全一致**:`@MinLength(8)` + `@MaxLength(128)` + `@Matches(/^(?=.*[a-zA-Z])(?=.*\d).+$/)` | `BAD_REQUEST=40000` |

**业务流程**(严格按评审稿 §5.2 顺序 1→5;不得调换 step 2 与 step 3,沿 §5.5 timing 防御):

1. `findFirst(notDeletedWhere)` 取当前 `passwordHash`;找不到 → `USER_NOT_FOUND=10001`
2. `bcrypt.compare(oldPassword, user.passwordHash)`;失败 → `OLD_PASSWORD_INVALID=10005`(**完整跑完**,禁止"先比对 `oldPassword === newPassword` 跳过 bcrypt"的优化)
3. 严格 `===` 比较 `oldPassword / newPassword`(密码大小写敏感、空白显著,**不** `trim` / `toLowerCase`);相同 → `NEW_PASSWORD_SAME_AS_OLD=10006`
4. `bcrypt.hash(newPassword, BCRYPT_SALT_ROUNDS=10)`
5. `prisma.$transaction` 内 `user.update + auditLogs.log('password.change.self', ...)` 原子(沿 emergency-contacts / certificates 范式)

**响应**:沿 `userSafeSelect`(10 字段),**永不**含 `passwordHash` / `deletedAt`。

### 4.2 新增 2 个 BizCode(沿 PR #117 / 评审稿 §5.3)

| code | name | HTTP | 触发场景 | 段位归属 |
|---|---|---|---|---|
| `10005` | `OLD_PASSWORD_INVALID` | 401 | `PUT /api/users/me/password` 时 `oldPassword` 与当前 `passwordHash` 不匹配 | 沿 100xx users 业务级,LOGIN_FAILED=10004 之后下一可用号位 |
| `10006` | `NEW_PASSWORD_SAME_AS_OLD` | 400 | `PUT /api/users/me/password` 时 `newPassword === oldPassword`(严格 === 比较) | 同上,再下一可用号位 |

**关键设计选择**(沿评审稿 §5.3):

- **`OLD_PASSWORD_INVALID` 不复用 `LOGIN_FAILED=10004`**:本人改密在**已登录态**,`currentUser` 已知,**无账号枚举攻击面**;前端拿到精确错误码可给"当前密码不正确"明确提示,优于"账号或密码错误"模糊提示
- **`NEW_PASSWORD_SAME_AS_OLD` 不复用 `BAD_REQUEST=40000`**:DTO 校验兜底是泛错;"新旧密码相同"是业务级语义校验,前端展示文案与一般 DTO 错误不同,值得独立码位

### 4.3 新增独立 throttler 实例(沿 PR #117 / 评审稿 §5.4)

- **throttler name**:`password-change`(在 `ThrottlerModule.forRootAsync` 中注册;与 `default`(LoginThrottle 用)物理隔离)
- **默认参数**:IP 维度 5 次 / 60 秒(`PASSWORD_CHANGE_THROTTLE_LIMIT` / `PASSWORD_CHANGE_THROTTLE_TTL_SECONDS` 可配)
- **新装饰器**:`@PasswordChangeThrottle()`(metadata 标记型;`PASSWORD_CHANGE_THROTTLE_KEY` 与 `PASSWORD_CHANGE_THROTTLER_NAME = 'password-change'` 常量同文件导出;沿 `@LoginThrottle` 范式)
- **`ThrottlerBizGuard` 改造**:`shouldSkip` 改"任一 metadata 命中即进入";`handleRequest` 按 `throttler.name` 与 metadata 对应关系判断,不匹配直接放过(物理隔离两个 throttler)
- **storage**:沿 V1.1 内存 storage,**不引入** Redis;不暴露 `Retry-After` / `X-RateLimit-*` 头(沿 V1.1 §17.7)
- **超限错误码**:沿现有 `TOO_MANY_REQUESTS=42900`(message: `请求过于频繁，请稍后再试`)

**关键铁律**:登录失败爆破**不**消耗改密配额,反之亦然(双 throttler 物理隔离)。

### 4.4 新增 audit event(沿 PR #117 / 评审稿 §5.6)

- **事件名**:`'password.change.self'`(沿现有 kebab-case `<resource>.<action>.<scope>` 风格;与 `profile.update.self` 对称)
- **写入位置**:`UsersService.changeMyPassword` 在 `prisma.$transaction` 内调 `auditLogsService.log({ event: 'password.change.self', actorUserId, actorRoleSnap, resourceType: 'user', resourceId, meta: auditMeta, tx })`
- **`AuditMeta`**:由 controller 层从 `@Req()` 构造,显式传 service(沿 D6 v1.1 §11.2 / D8 拍板;不引入 cls-rs / AsyncLocalStorage)
- **`AuditContext` 锁形**:`requestId / ip / ua` 3 必填(`requestId` 必为非空字符串;`ip` / `ua` 可为 null,但字段必须存在)
- **`UsersModule` 新增 `imports: AuditLogsModule`**(供注入 `AuditLogsService`)
- **敏感字段防御**:**禁止**写入 `oldPassword` / `newPassword` / `passwordHash` 任何明文或 hash;e2e 强断言序列化后的 audit 记录不含 `TEST_PASSWORD` / `NEW_PASSWORD` / bcrypt hash 前缀 `$2`

### 4.5 不主动吊销旧 token(沿 PR #117 / 评审稿 §5.7)

- **行为**:本人改密成功后,`JwtStrategy.validate` 仍仅看 `deletedAt + status === ACTIVE`,**不读** `passwordHash`,密码 hash 变化**不影响**已签发 token
- **设计依据**:沿 v1 §9 + [`security.md` Token 吊销升级路径](../security.md);**与管理员重置(`PUT /api/users/:id/password`)行为完全一致**
- **如需立即阻断**:管理员把目标用户 `status` 改 `DISABLED`(`JwtStrategy.validate` 每请求查库)
- **未来升级路径**:`tokenVersion` / refresh token / token revoke 归 **P0-E** 统一评审;**本接口不预实现**;e2e 反向断言锁定本行为(改密后旧 token 仍能 `GET /me` 返 200)

### 8.2 v0.13.0 段内新增 / 强化的红线(P0-D)

沿 P0-D 评审稿 §5 + `CLAUDE.md §9` 修订后内容:

- **本人改密接口路径锁死**:`PUT /api/users/me/password`(沿评审稿 §3.1);**禁止**改路径名(如 `POST /api/users/me/change-password`)/ HTTP method(如 PATCH / POST);**禁止**新增同义接口
- **入参严格白名单 2 字段**:`oldPassword` + `newPassword`;**禁止**接受 `username` / `email` / `role` / `status` / `passwordHash` / `id` / `confirmPassword` 任何额外字段(`forbidNonWhitelisted` 兜底)
- **业务流程顺序锁死**:1. 查 hash → 2. bcrypt.compare oldPassword → 3. 严格 `===` 比较 oldPassword/newPassword → 4. bcrypt.hash newPassword → 5. 事务内 update + audit log;**禁止**调换 step 2 与 step 3(timing oracle:先比较会泄漏"新密码是否等于旧密码"信息)
- **`oldPassword === newPassword` 严格 `===` 比较**:**不** `trim` / `toLowerCase`(密码大小写敏感、空白显著)
- **timing 防御:bcrypt.compare 必须完整跑完**:**禁止**"先比对 `oldPassword === newPassword` 跳过 bcrypt"的优化(避免泄漏"newPassword 与 oldPassword 是否相同"信息)
- **`OLD_PASSWORD_INVALID = 10005` 与 `LOGIN_FAILED = 10004` 严格分离**:本人改密**不**复用 10004 的模糊语义;前端可精确提示"当前密码不正确"
- **`NEW_PASSWORD_SAME_AS_OLD = 10006` 与 `BAD_REQUEST = 40000` 严格分离**:不复用 DTO 校验兜底
- **独立 throttler 实例 `password-change` 与 `default` 物理隔离**:登录失败爆破**不**消耗改密配额,反之亦然;**禁止**复用 `@LoginThrottle()` 装饰器(语义混淆)
- **限流参数不硬编码**:从 `app.config.ts.passwordChangeThrottle.{limit, ttlSeconds}` 注入;**禁止**写死 5 / 60 在装饰器里
- **audit event 严格命名 `'password.change.self'`**:沿现有 kebab-case `<resource>.<action>.<scope>` 风格;**禁止**改成 `UserPasswordChangedSelf`(PascalCase)/ `password.change`(省略 self)/ `user.password.changed.self`(冗长)
- **audit 不写敏感字段**:**禁止**写入 `oldPassword` / `newPassword` / `passwordHash` 任何明文或 hash;e2e 强断言序列化后的 audit 记录不含 `TEST_PASSWORD` / `NEW_PASSWORD` / `$2`(bcrypt 前缀)
- **改密后不主动吊销旧 token**(沿评审稿 §5.7);**禁止**"顺手加吊销逻辑";e2e 反向断言锁定本行为(改密后旧 token 仍能 `GET /me` 返 200);如改本行为,先改 [`security.md` Token 吊销升级路径](../security.md) + P0-E 评审
- **`UpdateMyProfileDto` 严格白名单**:仅 `nickname` / `avatarKey`;**禁止**接受 `oldPassword` / `newPassword` / `password` / `passwordHash` / `role` / `status` / `deletedAt` / `id` / `lastLoginAt` / `username` / `email`(沿 `CLAUDE.md §11` + users-me.e2e-spec.ts FORBIDDEN_FIELDS)
- **不改 `PUT /api/users/:id/password`(管理员重置)**:契约保持不变(`ResetUserPasswordDto { newPassword }`,无 `oldPassword`);P0-D 不接管理员重置接口
- **不引入 schema / migration / seed 变更**:本接口零 prisma 变更(沿评审稿 §4 严禁)

### 8.3 v0.13.0 段常见反模式提醒

| 反模式 | 为什么禁止 |
|---|---|
| "顺手"接管理员重置接口加 `oldPassword` 字段 | 改密语义已通过本人接口 `/me/password` 提供;`/:id/password` 是管理员重置,无需 `oldPassword`(沿评审稿 §1.3)|

### 4.4 3 个 AuditLogEvent(union 现 17 项)

| Event | 触发位置 | 关键 extra 字段 |
|---|---|---|
| `attachment.upload` | `attachments.service.create` | `operation` / `attachmentType` / `ownerType` / `ownerId` / `mime` / `size` / `scope`(`'self' \| 'other' \| null`)/ `ownerTable` |
| `attachment.delete` | `attachments.service.delete` | 上 + `deletedByPath`(`'owner' \| 'admin'`;按 `currentUser.id === uploadedBy` 判定) |
| `attachment.config.change` | 配置三表 11 个写端点共用 | `configType`(`'type' \| 'mime' \| 'sizeLimit'`)/ `operation`(`'create' \| 'update' \| 'update-status' \| 'delete'`)+ 业务字段(type: `code` / `ownerTable`;mime: `typeConfigId` / `mime`;size: `typeConfigId` / `maxSizeBytes`) |

### 4.5 20 条 `attachment.*` Permission seed(沿 D7 v1.0 §6.1 Q11 锁定)

- **member×8**:`attachment.{upload,view,update,delete}.member.{self,other}`(4 段 code;F1 / PR #70 适配)
- **certificate×8**:`attachment.{upload,view,update,delete}.certificate.{self,other}`
- **activity×4**:`attachment.{upload,view,update,delete}.activity`(粗粒度;Q10:无 self/other 区分)

**seed 幂等**(全部 upsert);**不自动给任何 user 分配 `member` 角色**(Q2 沿用)。

### 4.6 `member` 内置 RbacRole + 9 条 RolePermission

- `member` placeholder 角色(USER 内置;运营可重命名)
- 9 条 RolePermission:8 条 `.self`(member×4 + certificate×4)+ 1 条 `attachment.view.activity`
- **不创建 ADMIN 内置角色**(Q12 v1.0 沿用挂起;留独立专项评审 PR)
- **不接入 dept-chief / dept-deputy 层级**(seed 真实名留 `.env.seed.local`)

### 4.7 首次业务模块接入 `rbac.can()`(沿 D7 F3 + F5)

- attachments 主模块 7 个端点:**入口仅 `JwtAuthGuard`**(无 `@Roles(...)`)
- Service 层显式调 `await this.rbac.can(user, action, resource?)`,失败统一抛 `BizException(BizCode.RBAC_FORBIDDEN)`(30100)
- PermissionsModule **`exports: [RbacService]`** 首次外露给业务模块(本 PR #76 接入)
- scope 自动判 `.self` / `.other`:certificate Service 层先查 `Certificate.memberId` 转 RBAC `member` resource;activity 粗粒度(Q10)
- **读路径无权统一返 13001**(Q13 信息泄漏防御);**写路径无权返 30100**

### 4.8 audit 同事务 fail-fast(沿 D7 F6 + D6 R5)

- 主模块 `create` / `delete` wrap `$transaction(async (tx) => { tx.attachment.X(...); auditLogs.log({ tx }) })`
- 配置三表 11 个写端点同范式;P2002 兜底外层包 `$transaction`,失败 → BizException + 事务回滚 + audit 不写
- **校验链留事务外**(PR #6c Q7 拍板):减小事务体积,失败回滚廉价
- **不审计 PATCH metadata**(Q7 v0.2 锁;value 低)/ **不审计 view/list**(R4)/ **不审计失败操作**(F6 fail-fast)

### 4.9 多态外键 + ownerType 双层校验(沿 Q5 v1.0)

- 业务层 `AttachmentOwnerType` TS enum(`'member' | 'certificate' | 'activity'`):代码防错,编译期已知
- 配置表 `attachment_type_configs.code`:运行时权威白名单,运营可启停 / 新增
- 校验顺序:**业务层 enum 先**(便宜)→ **配置表权威**(运行时);两层都必须命中
- 两者必须保持同步:新增业务 ownerType 时同步改 enum + 配置表 seed

### 4.10 PII 检测(沿 Q9 v1.0)

- 身份证号正则 `\d{17}[\dXx]` 在 `originalName` / `description` / `tags` 三字段
- **不调用 OCR**;**不入库身份证号字符串**;只检测元数据文本
- 命中抛 `BizException(BizCode.ATTACHMENT_PII_DETECTED)`(13015)

### 4.11 系统级 MIME 黑名单(沿 Q13 + §6.6)

- 精确 8 项:`application/x-msdownload` / `application/x-executable` / `application/x-dosexec` / `application/x-sh` / `application/x-bat` / `application/zip` / `application/x-zip-compressed` / `application/x-rar-compressed`
- 通配 1 项:`video/*`(完整禁视频;走"独立多媒体管理评审")
- Service 层硬编码;**即使后台运营在 `attachment_mime_configs` 配置为 ACTIVE 也不允许通过**

### 4.12 信息泄漏防御(沿 Q13 v1.0)

- 读路径(`GET /api/v2/attachments/:id`):不存在 + 无权统一返 `13001 ATTACHMENT_NOT_FOUND`

## 9. 已知 quirk:`gh pr merge` worktree checkout main 冲突预案

本仓库在 `.claude/worktrees/` 下创建临时 worktree 时,`gh pr merge --squash --delete-branch` 会撞:

```
failed to run git: fatal: 'main' is already used by worktree at '/Users/.../srvf-nest-api'
```

**根因**:主仓 worktree 占用 `main` 分支,gh 试图本地 checkout main 失败。

**预案**(沿 PR #54-#80 累计 ≥ 14 次验证成功):

```bash
# 1. 仍执行 gh pr merge(GitHub 端会完成 squash merge)
gh pr merge <PR#> --squash --delete-branch
# → 报 worktree 错误,但 GitHub 端已 MERGED

# 2. 验证 GitHub 端 MERGED 状态
gh pr view <PR#> --json state,mergedAt,mergeCommit

# 3. 检查远端分支是否已删(--delete-branch 因 worktree 错误未执行)
git ls-remote --heads origin <branch-name>

# 4. 手动删除远端分支
git push origin --delete <branch-name>

# 5. 主仓库 ff main 到最新
git -C /Users/dengwang/Documents/coding/srvf-nest-api pull --ff-only origin main

# 6. worktree 同步到 main(开下一 PR 前)
git fetch origin main && git reset --hard origin/main
```

详细步骤见 v0.8.1 handoff §11(预案不变;PR #54-#80 全部按此流程操作)。

## 6. 批次 3 核心业务链路

```
Activity(draft → published)
  → ActivityRegistration(USER 自助 / ADMIN 代报名)
    → pass / reject / cancelled(partial unique 取消后允许重报;capacity 仅统计 pass)
      → AttendanceSheet(pending,事务内 create Sheet + N Records)
        → review-detail(R25 Activity+Sheet+Records[含 Member])
        → edit(D38 后端事务内生成 previousSnapshot + version+1;旧 records 软删 + 新 records 创建)
        → approve(R31 contributionPoints 必填;同事务触发 attendance.recorded)
        → reject(reviewNote 必填)
          → /me/attendance-records(Q-A14 仅 approved Sheet 内 records)
```

核心不变式(代码层强约束,**改 V2 代码前必须保留**):

- 同 memberId × `[checkInAt, checkOutAt)` 左闭右开,跨 Sheet / 跨 Activity 全局不重叠
- `serviceHours` 未传自动 `(checkOut - checkIn) / 3600`;`<= 0` 拒(`@Min(0.01)` + service 双兜底);`> 跨度` 拒(`22071`);允许 `< 跨度`(吃饭休息不计入)
- `registrationId !== null` 时校验 `registration.activityId === sheet.activityId`(`22073`,mismatch 与 not found 同码,沿 §1.7 风格)
- `attendance.recorded` 业务事件 **仅 approve 触发**,rejected / submit / edit / delete 一律不触发
- CSV 导出走 `StreamableFile`,`ResponseInterceptor` 已通过 `instanceof` 自动跳过,**未改 interceptor**;XLSX 入参直接 400

### 12.1 状态机升级:3 态 → 5 态(沿 D-S6)

### 12.2 `approved` 语义升级(沿 D-S7)

- **v0.4.0 之前**:`approved` = "APD approve 后即 approved"(单级审核)

### 12.4 D14 ContributionRule 系统预填(沿 D-A8 候选 5.B)

- 触发点:`POST /api/v2/activities/:activityId/attendance-sheets` **事务内**
- 匹配维度:`(activityTypeCode, attendanceRoleCode, durationMinutes)`(`durationThreshold` 在 service 端按 `record.serviceHours` 比较,**不在 where 子句**)
- 预填规则:
  - `record.contributionPoints !== null`(调用方传值)→ **不覆盖**
  - `record.contributionPoints === null` + 匹配规则 → 按规则预填(`serviceHours <= durationThreshold` 取 `pointsBelow`;否则取 `pointsAbove ?? pointsBelow`)
  - **无匹配规则** → `contributionPoints` 保持 `null`,**不抛错**(沿 D-S11 `22048` 不开)
- 每日上限:`rule.dailyCap` 默认 1.5(沿 Q-OPEN-7 锁定);最终值 = `MIN(candidatePoints, effectiveCap)`
- **入参语义**:`AttendanceRecordInputDto.contributionPoints` 是"**可省略**"(`@IsOptional`),**不是** OpenAPI `nullable: true`;传 `null` 会被 `@IsNumber` 拒
- **不暴露** `ContributionRule` CRUD 接口(留运营后台或单独管理 PR)
- **不引** `contribution_points` 独立流水表 / cron-job(D49 / R32 永久不做,沿 v0.4.0)

### 12.5 ContributionRule NULL `durationThreshold` 行为(已知设计限制)

- partial unique index `contribution_rules_activity_role_threshold_active_unique`(`WHERE deletedAt IS NULL AND status = 'ACTIVE'`)在 PostgreSQL NULL 语义下**不阻止** `durationThreshold = NULL` 多条 ACTIVE 并存
- service 兜底:`tx.contributionRule.findMany({...}).orderBy({ createdAt: 'asc' })`,取第一条
- 后续运营后台落地时**需要补**:更严格的创建 / 更新校验(同维度同 NULL threshold 不允许多条 ACTIVE)或告警机制
- 当前批次只取首条**不阻塞创建**;e2e `D14 ContributionRule 预填 → NULL durationThreshold 多条 ACTIVE 规则 → 按 createdAt ASC 取首条` 已覆盖

### 12.6 D11 Activity.completed 推动(沿 D-S10 / 业务规则文档 §3)

- 触发点:首张 `AttendanceSheet` 提交时,**事务内** `Activity.statusCode published → completed`
- **不等待** `final-approve`;**不在** APD 一级或终审任意环节推动
- 后续 Sheet 提交幂等(已 `completed` → 无操作)
- `reject` / `final-reject` **不回退** `Activity.completed`
- **`completed` 在批次 4 的业务语义** = "活动已进入考勤提交阶段"(至少有一张 Sheet 已提交),**不**代表"全部终审通过"
- 业务规则文档 §3.4:运营可通过 `AttendanceSheet` 列表观察"虽 `completed` 但无通过考勤"

### 12.7 权限边界:终审权限当前沿用 ADMIN / SUPER_ADMIN(沿 D-S2)

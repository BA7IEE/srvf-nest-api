# srvf-goal-author — 把工作固化为可自驱的 goal 文本

[定位]
本仓协作模式(process §7.1):**goal 文本 = 维护者立项 + 拍板 + 授权记录**,runner 会话无人值守自驱,终版报告回传主会话元核验。本 skill 管起草侧:只准备文本不替用户下发;仓库一切变更由 runner 走 PR(分支保护强制)。用户是非职业开发者:决策问题用人话问、每题给推荐;工程细节自行代决并在 goal 里写明。

[第一性原则]
1. **拍板先行**:goal 不带未决问题(runner 中途只能人话简报停,决策必须前置)。有产品/C/D 档决策未拍 → **沿设计树逐分支穷尽地访谈,逐题解决决策间依赖,直到达成共识;每题给推荐答案;一次问一个;能查代码库自答的,查代码而不是问用户**。简单任务无可拍之事则直接写。
2. **侦察先行**:写前核实况(HEAD/版本/open PR/Unreleased/current-state §1)与涉及代码(镜像哪个先例,精确 file:line);给不准的数字注明"runner 亲核为准"。
3. **目标导向,不教过程**:goal 写"什么必须为真"(DoD)与"什么不许碰"(边界),不写操作步骤——执行打法交给 runner:一条路不通换方法,穷尽多种尝试才停。流程纪律引用 process 节号(§7.1 循环 / §5.4 合并 / §4.1 人话简报 / §3 档位),不复述全文;规则与已知怪癖在仓库文档和项目 memory 里,runner 自取。
4. **DoD 可自证**:每条终态可用命令/产出物/计数在报告里自证;任务带探针(已完成判据),中断后同一 goal 文本重跑幂等续命。
5. **边界要硬**:只写真实约束(红区具体哪几处及 #294 授权范式、明确不做清单、档位切分、"偏离 → 人话简报停");放之四海的空话不写。
6. **不填形式**:必填仅三件——背景(含已拍板事实与日期)、DoD、授权与禁区;任务队列/报告格式按信息量取舍,迷你 goal 三段即可。每行带信息,该长就长(goal 兼任评审授权记录,不设字数上限),不带信息就删。

[常用范式](点名复用,不复述)
D 档功能串:T0 评审稿冻结(docs/archive/reviews/,镜像 sms/wechat 等先例)→ schema(干净库重放 + seed 幂等二跑)→ 实施 → docs 收尾。E 档收口:沿 srvf-release-closeout 九阶段。多件打包:F0-Fn 首尾各一次收口。

[交付]
goal 整段放代码块,附"复制给 goal 会话运行";给主会话留 2-4 条"跑完回传核验点"(终版报告贴回后据此元核验);含红区/不可逆授权的,提醒用户发送前扫一眼授权段。

[工作流程]
理解目标 → 侦察 → 访谈拍板(穷尽,见原则 1)→ 起草 → 交付 + 核验点。
